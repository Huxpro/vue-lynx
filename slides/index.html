III · Case 2 · Elk · draggable sheet

One sheet, three owners.

webnative
① <Transition> · open / close
② MTS worklet · drag · rubber
③ <scroll-view> · scroll ↔ drag

The drag runs on the main thread — so even while Vue's background thread fetches, diffs, or rebuilds the list, the sheet still tracks the finger. On the web, gesture JS and render share one thread.

The draggable sheet — the hard part isn't sliding up, it's three state machines cooperating. Vue Transition owns whether the sheet exists in the visual state (open/close). A Main-Thread Script worklet owns where the finger is each frame (drag + rubber-band). The native <scroll-view> owns who this touch belongs to (scroll vs sheet). Draw the boundaries and you get Vue's declarative state, native scrolling, and near-native tracking at once.

Structure = property ownership. sheet-surface-transition is owned by Vue Transition (CSS classes); the inner sheet-surface is owned by the drag (inline transform). If both wrote the same element's transform, the inline style clobbers the Transition class — the classic "animates once, then never again" bug.

The dual-thread payoff: on the web, gesture JS, Vue render, and business code share one main thread — any long task drops frames. Here the drag lives on the UI thread, next to native input and layout, so it stays glued to the finger regardless of what Vue is doing.

III · Case 2 · Elk · draggable sheet

Three owners, in code.

Sheet.vue · template 
<!-- ① Vue owns open / close -->
<Transition name="sheet">
  <view v-show="open"
        class="sheet-surface-transition">
    <!-- ② MTS owns the drag -->
    <view ref="surface" class="sheet-surface"
          :main-thread-bindtouchmove="onMove">
      <!-- ③ native scroll -->
      <scroll-view ref="panel"
        :main-thread-bindscroll="onScroll" />
    </view>
  </view>
</Transition>
onMove · 'main thread' 
function onMove(e) {
  'main thread'
  // follow the finger — no Vue render
  surface.setStyleProperty(
    'transform', `translateY(${y}px)`)

  // arbitrate: list at top → sheet wins
  if (deltaY > 0 && scrollTop <= 0)
    panel.setAttribute('enable-scroll', false)
}

// settle done → hand back to Vue
runOnBackground(requestClose)()

Minimal, representative slice. ① The <Transition> + v-show pair keeps the node mounted (scroll position and state survive close) while running the full persisted-transition lifecycle — the capability that let Vue Lynx graduate this from experimental. ② The drag is a 'main thread' worklet bound via :main-thread-bindtouch*; it writes transform/opacity with setStyleProperty, bypassing Vue diff. ③ Gesture arbitration: an 8px lock decides direction/source once, then — if you're dragging down with the list already at the top — the sheet disables the native scroll (enable-scroll=false) and takes the drag; otherwise it defers to the scroll-view. runOnBackground only crosses threads at the state boundary (on close), never per frame.

Why the web struggles: reading live scrollTop, arbitrating ownership, smoothing velocity, applying rubber resistance, and toggling inner scroll all need the JS main thread — which the browser shares with render. Lynx runs that logic on the UI thread instead.

The engineering chapter. Everything you just saw was built in two weeks by one person — and this chapter is the honest answer to "how". IV · Runtime
Background thread
Main (UI) thread
Vue 3 runtime · VDOM @vue/runtime-core · untouched

Vue runs whole — on the background thread.

First decision: which thread does Vue live on? An early community experiment put Vue on the main thread — every event then had to be forwarded across. But Lynx delivers events to the background thread natively, so we run the entire runtime there: reactivity, diff, lifecycle, your handlers. Not a fork of Vue — the actual @vue/runtime-core, untouched.

createRenderer() — a renderer, not a fork.

Vue's official custom renderer API is the whole trick. createRenderer() wants synchronous nodes — parentNode(), nextSibling() must answer immediately, but the real elements live on another thread. So nodeOps dual-writes: it maintains a lightweight ShadowElement linked-list on the background thread (satisfying every synchronous read Vue makes) while queueing the real work for later. Same pattern ReactLynx uses for its snapshot instances.

Events ride back up — handlers never leave Vue.

The return path closes the loop. Functions can't cross threads, so they never do: each handler registers under a numeric sign; the main thread dispatches the sign, the background thread looks it up and calls your Vue function right where it lives — closures, reactivity and all. That's why every Vue semantics survives: nothing that matters ever crossed.

Lynx's deliverable is one bundle carrying two programs — background code for the JS VM, main-thread code for Lepus (PrimJS). Two VMs, two entries, one artifact; the same file renders natively and, through Lynx for Web, in the browser. So the toolchain question becomes: how does one build emit both worlds from one Vue codebase?

The same code enters twice — webpack layers route it.

Both entries import your actual app. Each entry is compiled under a webpack layer — vue:background and vue:main-thread — and issuerLayer rules give each layer its own loader chain: the BG side compiles the full SFC; the MT side extracts only what the main thread needs. Per-entry scoping falls out of webpack's dependency graph for free. (We adopted this layer approach from ReactLynx after our first flat-bundle architecture couldn't isolate multi-entry apps.)

IV · Toolchain

We wrote a plugin — not a compiler.

Rspeedy Rspack / Rsbuild rspack-vue-loader PostCSS

The reuse story, toolchain edition. Lynx's toolchain (Rspeedy) is Rspack/Rsbuild — and the Vue ecosystem already runs on Rspack, so the entire SFC pipeline came off the shelf: rspack-vue-loader, PostCSS, HMR. pluginVueLynx() is a thin adapter: split each entry into the two layers, wire the worklet loaders, configure CSS — that's the whole surface. Framework-agnostic isn't a slogan; it's measured in how little we had to write.

Your CSS isn't translated. It's just CSS.

The quiet superpower behind the whole styling chapter: Lynx ships a real CSS engine on the native side — selectors, cascade, animations. So <style scoped> maps to Lynx's cssId scoping, CSS modules and imported stylesheets pass straight through, v-bind() in CSS rides CSS variables, and Tailwind works because PostCSS is just PostCSS. No StyleSheet-object dialect, no translation layer to leak.

Main (UI) thread
onScroll() a Vue handler
↑ event ↓ ops 2 crossings behind your finger

The dual-thread cost: gestures wait for the round trip.

Honest about the cost before selling the reward. A scroll-follow animation the normal way: the event fires on the main thread, hops to the background for your handler, and the resulting ops hop back — two crossings on every frame of a gesture. On a busy page that's visible lag. This is the one place the dual-thread architecture charges you.

IV · Main-Thread Script
Background thread
Main (UI) thread
onScroll() 'main thread'
0 crossings — it runs where events fire

One directive. The function changes threads.

Watch the block move. Add 'main thread' as the first line and the compiler relocates that function into the main-thread bundle — it now runs synchronously where events fire, with direct element access. The rest of your component stays ordinary Vue. That's the progressive-enhancement shape again: the dual-thread reward stays, the cost becomes opt-out per function.

Draggable.vue 
<script setup>
import { useMainThreadRef } from 'vue-lynx'
const box = useMainThreadRef()

const onScroll = (e) => {
  'main thread'
  box.current?.setStyleProperty('transform',
    `translateX(${e.detail.scrollLeft}px)`)
}
</script>

<template>
  <scroll-view :main-thread-bindscroll="onScroll" />
  <view :main-thread-ref="box" />
</template>

Still a .vue file. Still ref-shaped. Zero-latency.

The whole MTS surface in one file: 'main thread' marks the function, main-thread-bind* attaches it to the event, useMainThreadRef() gives synchronous element access — setStyleProperty lands the same frame the finger moves.

IV · The landscape

Zoom out: every stack is layers — and seams.

The promised deep-dive. In the prologue we speed-dated the candidates by feel; now, with the runtime, toolchain and MTS on the table, we can afford the architecture-level version. Split any cross-platform stack into five layers; between them sit four seams — each one either an extension point or a welded wall. Let's grade everyone honestly, Lynx included.

IV · The landscape · the map
open — an extension point half-open welded shut

The map. Split any cross-platform stack into five layers; between them sit four seams, each a yes/no question: EP1 plug in any frontend? EP2 swap the render model? EP3 add native capabilities? EP4 reach a new platform? An open seam is an extension point; a welded seam is a wall.

IV · The landscape

Web — the DX everyone wants. But the sandbox caps how native it can ever feel.

The Web: EP1 wide open — any framework. But rendering and capabilities are welded shut by the sandbox; a new platform just means "a browser exists there". Great DX, capped UX.

IV · The landscape

Ionic — the Web in a shell. Not native enough.

Ionic (and every WebView shell): keeps the Web's openness on top, Capacitor pries the capability seam half-open — but rendering is still the DOM in a WebView. It inherits the Web's ceiling: not native enough.

IV · The landscape

NativeScript — direct native access. Not Web enough.

NativeScript: the opposite trade. The most direct native access of anyone — but it isn't really the Web: no real CSS engine or DOM semantics, rendering and new platforms welded. Not Web enough.

IV · The landscape

Flutter — Dart plus self-rendering. A parallel universe.

Flutter: opens the bottom half wide — self-renderer, embedder for new platforms. The price: the only door on top is Dart. It doesn't extend the Web; it replaces it. A parallel universe.

IV · The landscape

React Native — closest to the dream. But the frontend seam is React-only.

React Native: capabilities open, huge ecosystem — closest to the dream. But the frontend seam is welded to React: Metro, the renderer, the mental model. If you're Vue, this door was never yours.

IV · The landscape · the answer

Lynx — all four seams open. Exactly the door Vue was waiting for.

The answer. EP1: Rspack-based toolchain, framework-agnostic — real CSS, any frontend. EP2: the engine adapts native primitives or self-renders. EP3: Native Modules. EP4: mobile + desktop shipped, TV/VR run on the self-renderer. Four seams, none welded — including the one Vue needs.

IV · The landscape

Lynx evolved framework-agnostic.

Lynx started with ReactLynx, but the platform was deliberately evolved to be framework-agnostic — the frontend layer is a genuine extension point, not a React feature. That's not marketing; you'll see the receipts in a second.
