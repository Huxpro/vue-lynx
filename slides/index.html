onMove · 'main thread' 
function onMove(e) {
  'main thread'
  surface.setStyleProperty('transform', `translateY(${y}px)`)
  if (deltaY > 0 && scrollTop <= 0)   // at top → sheet wins
    panel.setAttribute('enable-scroll', false)
}

The draggable sheet — the hard part isn't sliding up, it's three state machines cooperating. ① Vue <Transition> owns whether the sheet exists in the visual state (open/close); v-show keeps the node mounted so scroll position and state survive close, while the persisted-transition lifecycle runs — the fix that let Vue Lynx graduate this from experimental. ② A 'main thread' worklet owns where the finger is each frame: bound via :main-thread-bindtouch*, it writes transform with setStyleProperty, bypassing Vue diff, plus rubber-band resistance past the top. ③ The native <scroll-view> owns the touch: an 8px lock decides direction/source once, then if you drag down with the list already at the top the sheet disables native scroll (enable-scroll=false) and takes the drag; otherwise it defers. runOnBackground crosses threads only at the state boundary (on close), never per frame.

Structure = property ownership: the outer surface is owned by Transition (CSS classes), the inner by the drag (inline transform) — write both to one element and inline clobbers the class, the classic "animates once, then never again" bug.

Dual-thread payoff: the drag lives on the UI thread, next to native input and layout, so it stays glued to the finger even while Vue's background thread fetches / diffs / rebuilds the list. On the web, gesture JS and render share one thread — any long task drops frames.

And how was this built?

The engineering chapter. Everything you just saw was built in two weeks by one person — and this chapter is the honest answer to "how". Three adaptations, each revealing one piece of Lynx's architecture: split Vue across two threads without breaking its semantics; make one toolchain emit both worlds; then make the main thread itself programmable. An AI harness threads through the whole story.

Main (UI) thread
event
your JavaScript
paint

The web runs on one thread — event, your JS, paint — and it all has to land inside a frame. Cross-platform pins native UI here too; pile an app on top and the budget blows.

Start from what everyone knows. The web is single-threaded: layout, paint, gestures, and all your JS share one thread. Cross-platform systems inherit a harder constraint — native UI must live on the main (UI) thread. Stack a framework's reactivity and diffing on top of that and the UI thread starves. That's the squeeze every cross-platform runtime has to answer, and Lynx answers it with a second thread.

Vue drives native elements through the Element PAPI. Run it here and reactivity, diff, your handlers all fight the UI — too much for one thread.

The naive shape. Vue could talk straight to Lynx's Element PAPI — __CreateView, __SetAttribute… — to build native UI. But that puts the entire Vue runtime on the same thread as layout and paint. It works, and it janks. So the real question is: how do we get Vue off the UI thread without losing its grip on the elements?

Lift Vue onto a background thread. The runtime is off the UI now — but how does it drive elements it can no longer touch?

Second decision: which thread does Vue live on? Lynx delivers events to the background thread natively, so we run the entire runtime there — reactivity, diff, lifecycle, your handlers. Not a fork; the actual @vue/runtime-core, untouched. But Vue's renderer wants synchronous DOM nodes, and the real elements are a thread away. That's the gap the next slide closes.

A ShadowElement tree fakes the DOM for Vue; edits leave as a flat ops buffer the main thread replays into Element PAPI.

Read it as a timeline, left to right. Vue's official createRenderer() wants synchronous nodes, so nodeOps maintains a lightweight ShadowElement linked-list on the background thread — satisfying every synchronous read Vue makes — while queueing the real work. Updates cross the boundary as a flat ops array (numbers and strings, one send per tick), and the main thread replays them straight into the Element PAPI. No interpreter box worth drawing: the replay is a switch loop. VDOM → ShadowElement → ops → PAPI → Native UI, one straight line.

Events ride back by numeric sign — handlers never leave Vue. One full lap across both threads is exactly one nextTick().

The line closes into a ring. Functions can't cross threads, so they never do: each handler registers under a numeric sign; the main thread dispatches the sign, the background thread looks it up and calls your Vue function right where it lives. That closes the loop — and one full lap of this ring is one nextTick(), the same mental model as web Vue, stretched across a thread. It's the one place the threading ever leaks into your code: onMounted(() => nextTick(() => lynx.createSelectorQuery()…)).

The map. Split any cross-platform stack into five layers; between them sit four seams, each a yes/no question: EP1 plug in any frontend? EP2 swap the render model? EP3 add native capabilities? EP4 reach a new platform? An open seam is an extension point; a welded seam is a wall.

Web — the DX everyone wants. But the sandbox caps how native it can ever feel.

The Web: EP1 wide open — any framework. But rendering and capabilities are welded shut by the sandbox; a new platform just means "a browser exists there". Great DX, capped UX.

Ionic — the Web in a shell. Not native enough.

Ionic (and every WebView shell): keeps the Web's openness on top, Capacitor pries the capability seam half-open — but rendering is still the DOM in a WebView. It inherits the Web's ceiling: not native enough.

NativeScript — direct native access. Not Web enough.

NativeScript: the opposite trade. The most direct native access of anyone — but it isn't really the Web: no real CSS engine or DOM semantics, rendering and new platforms welded. Not Web enough.

Flutter — Dart plus self-rendering. A parallel universe.

Flutter: opens the bottom half wide — self-renderer, embedder for new platforms. The price: the only door on top is Dart. It doesn't extend the Web; it replaces it. A parallel universe.

React Native — closest to the dream. But the frontend seam is React-only.

React Native: capabilities open, huge ecosystem — closest to the dream. But the frontend seam is welded to React: Metro, the renderer, the mental model. If you're Vue, this door was never yours.

Lynx — all four seams open. Exactly the door Vue was waiting for.

The answer. EP1: Rspack-based toolchain, framework-agnostic — real CSS, any frontend. EP2: the engine adapts native primitives or self-renders. EP3: Native Modules. EP4: mobile + desktop shipped, TV/VR run on the self-renderer. Four seams, none welded — including the one Vue needs.

Lynx started with ReactLynx, but the platform was deliberately evolved to be framework-agnostic — the frontend layer is a genuine extension point, not a React feature. That's not marketing; you'll see the receipts in a second.

static dynamic (the hole) a name that crosses the wire

Once pointers can't cross, names take over — and there are two ways to hand them out. Vapor names densely: a preorder walk gives every node a uid (2–7). Both threads run the same walk, so they agree on every name with zero bytes transferred — the addressable set is open (anything might get touched by an effect, event, or ref).

Element Templates name sparsely: the VDOM compiler already proved a closed set of dynamic points (the holes), and diffing guarantees the static nodes are never addressed — so only the holes get names and the grey nodes are invisible to the protocol. Sparse isn't saved, it's proven. Same template, two coordinates on the design space.

One template — a whole spectrum of renderers.

Even Vapor stays retained — it keeps pointers. But cross a thread up front and pointers can’t follow — so Lynx lives at the write-only end.

The design space is one wide spectrum — from fully retained to fully immediate. React VDOM is fully dynamic: interpret a tree and diff it every update. Vue's compiler-hinted, block-based VDOM does better — blocks let it skip the static subtrees. Vapor looks like it compiled everything away, but it still reserves a slice for the in-browser runtime: pointers, used to read / address live nodes. So Vapor is still retained mode.

Why that matters at a thread boundary. Retained pointers only survive a thread hop if the tree itself is shared: RN's Fabric keeps pointers in a C++ shadow tree that both sides reach. Lynx can't — we cross threads from the very first frame. Element Templates go the other way: write-only, one direction, much closer to an immediate-mode display list. That crosses threads beautifully — but the price is giving up dynamism on the MT side. Like a painting: you draw it once, then let the canvas go.
