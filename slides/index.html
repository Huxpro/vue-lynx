Case 1 · the headline delta
lib/stream-transport.ts 
// Web has a real ReadableStream → consume SSE incrementally.
// Native fetch can't stream a body → start ?mode=poll,
// then GET cursor pages until done. Same wire protocol.
export function selectStreamTransport(env) {
  return env.nativeLynx && !env.standardStreamingFetch
    ? 'poll'
    : 'sse'
}

@ai-sdk/vue's useChat assumed browser streaming. A custom composable keeps its exact surface — and picks the transport by capability.

The #200 story. Native fetch buffers an SSE body instead of streaming it, so the app detects the platform and switches to a poll transport — same server, same protocol, drop-in replacement for useChat with the identical { messages, status, sendMessage, stop } API.

Real apps hit real edges.

The recipe

The Vue stays Vue. The edges are the work.

Chapter V

The Lynx
platform itself.

Web-friendly on the surface, architecturally serious underneath.

Setup for the Lynx pitch. "Vue Lynx is only as good as the platform it sits on." Two-part argument: web-friendly surface, performant core.

The landscape

Every stack is layers — and seams.

Split a cross-platform stack into five layers. What decides openness isn't any one layer — it's whether the seam between them stays open (an extension point) or gets welded into a wall.

The frame. Don't grade a framework layer by layer — grade the seams. An open seam is an extension point; a welded seam is a wall. We'll count who leaves the most seams open.

The landscape · the map
open — an extension point half-open welded shut

The map. Five layers, top to bottom. Four seams between them, each a yes/no question: EP1 plug in any frontend? EP2 swap the render model? EP3 add native capabilities? EP4 reach a new platform?

The landscape

Web — open on top, sealed below. Takes any frontend, but the sandbox welds rendering and capabilities shut.

Web: EP1 open (any framework), but EP2/EP3 sealed by the sandbox; EP4 only half — you get a browser, not the metal.

The landscape

React Native — capabilities wide open, but the frontend is bound to React and each new platform is ported by hand.

RN: EP3 open (Native Modules), but Metro only speaks React (EP1 half), render swap needs a fork (EP2 half), new platforms are per-target (EP4 half).

The landscape

Flutter — the bottom half is wide open, at the cost of a single Dart-only door on top.

Flutter: EP3/EP4 open (self-render + embedder), but EP1 welded to Dart, EP2 half (fixed engine).

The landscape

NativeScript — the most direct native access, but rendering and new platforms are both welded shut.

NativeScript: EP3 open (direct native, no bridge), but EP2/EP4 sealed, EP1 half (not really the Web).

The landscape · the answer

Lynx is the only column with all four seams open — every layer leaves an extension point.

The answer. EP1: Rspack is framework-agnostic → React/Vue/Svelte + real CSS. EP2: the engine adapts native primitives or self-renders. EP3: Native Modules. EP4: desktop shipped, TV/VR/wearable already run on the self-renderer. Four seams, none welded.

The landscape

Four seams. Lynx welds none.

Framework-agnostic by design — the Web's developer experience, native's user experience, and room to reach any platform. That's the opening for Vue.

The payoff. Because no seam is welded, Vue can slot into EP1 and drive the whole stack. Web DX + Native UX. This is why Vue Lynx is even possible.

b · the runtime

Vue mutates. Native renders.

At runtime, Vue never touches a native view directly. It mutates a shadow tree; the changes travel as ops and are replayed on the UI thread.

Run-time story. The chain, built up next: nodeOps → shadow tree → ops → the bus → applyOps → Element PAPI → native. This is why gestures stay smooth: Vue's work and the UI thread's work are decoupled by a buffer.

Vue's createRenderer(nodeOps) — the same extension point React Native and others use. nodeOps is the Vue→Lynx adapter, the heart of it.

nodeOps mutates a ShadowElement tree — a lightweight mirror on the background thread — and appends each change to a flat ops buffer.

The shadow tree lets Vue call parentNode()/nextSibling() synchronously; the real elements live only on the main thread. Each node holds just an id — the cross-thread handle. Ops are a flat, JSON-serializable array keyed by that id.

applyOps replays each op through the Element PAPI — __CreateView, __SetAttribute, __AddEvent — then __FlushElementTree() commits to native.

The Element PAPI is Lynx's low-level element interface — the same one React Lynx targets. One commit per batch. Events flow back the other way: registered by sign, delivered to the BG handler.

b · the runtime · in the renderer
runtime/src/node-ops.ts 
export const nodeOps = {
  createElement(type) {
    const el = new ShadowElement(type)
    pushOp(OP.CREATE, el.id, type)   // → flat ops buffer
    scheduleFlush()                    // batch, one call per tick
    return el
  },
  // insert, patchProp, remove … each: mutate shadow, pushOp
}

Fourteen methods. Each mutates the shadow tree and pushes one op. That's the entire Vue→native adapter.

The whole surface is small — createElement, insert, patchProp, remove, setElementText, plus tree walkers. On the main thread, OP.CREATE becomes __CreateView; the batch ends with __FlushElementTree().

b · the trade-off

Why split at all?

What you win
  • UI thread free for 60fps gestures
  • Reactivity can't jank the scroll
  • One bundle, no native rebuild
  • Batched ops, no per-node bridge
What you pay
  • Reads across threads are async
  • No synchronous DOM-style measure
  • Truly-instant work → Main-Thread Script

Honest trade-off. The buffer that keeps gestures smooth also means you can't synchronously read layout from the background thread — you await it, or drop the hot code onto the main thread as a worklet. It's the same bargain React Lynx makes; the win is worth it.
