I · The gap

Cross-platform has many answers…

The field. The Web itself, React Native, Flutter, Ionic, NativeScript… cross-platform is a crowded, thirty-year-old conversation.

I · The gap

…but Vue never got a dominant one.

The gap. React has React Native. Flutter has Dart. Vue — with one of the largest frontend communities on earth, especially here — never got a native story it could call home. Weex tried. NativeScript-Vue tried. Nothing stuck.

Let's see why, structurally.

I · The gap · the map
open — an extension point half-open welded shut

The map. Split any cross-platform stack into five layers; between them sit four seams, each a yes/no question: EP1 plug in any frontend? EP2 swap the render model? EP3 add native capabilities? EP4 reach a new platform? An open seam is an extension point; a welded seam is a wall.

I · The gap

Web — the DX everyone wants. But the sandbox caps how native it can ever feel.

The Web: EP1 wide open — any framework. But rendering and capabilities are welded shut by the sandbox; a new platform just means "a browser exists there". Great DX, capped UX.

I · The gap

Ionic — the Web in a shell. Not native enough.

Ionic (and every WebView shell): keeps the Web's openness on top, Capacitor pries the capability seam half-open — but rendering is still the DOM in a WebView. It inherits the Web's ceiling: not native enough.

I · The gap

NativeScript — direct native access. Not Web enough.

NativeScript: the opposite trade. The most direct native access of anyone — but it isn't really the Web: no real CSS engine or DOM semantics, rendering and new platforms welded. Not Web enough.

I · The gap

Flutter — Dart plus self-rendering. A parallel universe.

Flutter: opens the bottom half wide — self-renderer, embedder for new platforms. The price: the only door on top is Dart. It doesn't extend the Web; it replaces it. A parallel universe.

I · The gap

React Native — closest to the dream. But the frontend seam is React-only.

React Native: capabilities open, huge ecosystem — closest to the dream. But the frontend seam is welded to React: Metro, the renderer, the mental model. If you're Vue, this door was never yours.

I · The gap · the answer

Lynx — all four seams open. Exactly the door Vue was waiting for.

The answer. EP1: Rspack-based toolchain, framework-agnostic — real CSS, any frontend. EP2: the engine adapts native primitives or self-renders. EP3: Native Modules. EP4: mobile + desktop shipped, TV/VR run on the self-renderer. Four seams, none welded — including the one Vue needs.

I · The gap

Lynx evolved framework-agnostic.

Lynx started with ReactLynx, but the platform was deliberately evolved to be framework-agnostic — the frontend layer is a genuine extension point, not a React feature. That's not marketing; you'll see the receipts in a second.

I · The gap
Web DX. Native UX.

The thesis, one line. Lynx gives you the Web's developer experience with native's user experience. And because the frontend seam is open — this is Vue's opening. Which brings us to…

II · Vue coverage · 6/9

provide / inject

App.vue · DeepChild.vue 
// App.vue — provide once
const theme = ref('light')
provide('theme', theme)

// any descendant — inject anywhere
const theme = inject('theme')

Dependency injection across the tree — reactive values included. Toggle the theme at the root; deep children re-render through the injected ref. This is also the mechanism Pinia and Router ride on, which is why they work.

III · Case 1 · AI Chat

Then it went native.

The keyboard slides in — the thread glides up with it. Hit send — the bubble flies from the prompt box into the conversation.

keyboard avoidance send motion MTS pressables

What changed — the upgrades. ① Keyboard avoidance: the native keyboardstatuschanged event drives setNativeProps transforms, so the prompt box and thread follow the keyboard with a native curve — no web viewport hacks. ② Send motion: your message physically detaches from the input and flies into the thread as a bubble. ③ Press feedback runs as Main-Thread Script.

Do live: focus the input (keyboard slides, layout follows), then send a message — watch the bubble.

III · Case 2 · Elk

Elk — a production-scale fork.

masto.js client content pipeline theme system domain logic

Elk's framework-agnostic layers, reused. Only the UI was rebuilt.

Scale is the argument here. Elk is a real Mastodon client, not a demo: timelines, threads, profiles, polls, content warnings, custom emoji, search, compose. We forked it and kept its brains — the masto.js API client, the Mastodon-HTML content pipeline, themes, domain logic — and rebuilt only the view layer on native elements. That's the porting economics: the bigger the app, the more carries over.

One minute of architecture — how all of this stays smooth. The Vue runtime — reactivity, diff, lifecycle, handlers — lives on the background thread. Next slide: the main thread lands beside it.

How it works
Background thread

Vue 3 runtime & your code

  • reactivity & lifecycle
  • component tree diff
  • event handlers
  • data fetching
ops → flat buffer ← events
Main (UI) thread

Native elements & render

  • layout & paint
  • gestures & scrolling
  • Main-Thread Script
  • ops interpreter

One bundle, two layers — no JS bridge round-trip on the hot path.

They talk via a flat ops buffer, batched per Vue tick. No JS bridge on the hot path — that's why the keyboard, the sheet, the swipes stay 60fps while Vue reactivity does its job. The kicker: Main-Thread Script — the sheet gesture, the pressables — is your Vue code, compiled onto the UI thread.
