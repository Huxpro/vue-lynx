IFR architecture scale trends

Five product architectures on one generated content SFC, scaled from ~1k to ~30k elements. Lead chart is cost space (MT gzip × FCP); FCP vs N follows. See also the Unified Benchmark Matrix for the same-host IFR × VDOM/Vapor × React story.

Linear scale — cost space + FCP vs N

Paint vs MT load cost (linear)

Linear cost space (playground-style, zero baseline). Same points as the framework bench's create×update chart: architectures that pay MT gzip without buying FCP shoot right; those that scale poorly shoot up. CPU ×1.

FCP vs content scale (linear)

Linear, zero baseline, 1k→30k — absolute FCP gaps. VDOM+IFR without ET crosses above plain VDOM and can DNF on deep MT mount. CPU ×1.

Log-log scale

Paint vs MT load cost (log)

Log-log cost space (playground-style). Each polyline is one architecture through 1k→30k. Right = heavier main-thread bundle; up = slower FCP. CPU ×1. Lower-left dominates.

FCP vs content scale (log)

Log-log. Same generated SFC at ~1k→30k elements. Lynx for Web, CPU ×1. α ≈ 1 → linear in N; α ≪ 1 → dominated by fixed costs.

Scaling exponents

α ≈ 0 → FCP dominated by fixed overhead (bundle parse / boot / IPC). α ≈ 1 → cost grows with content.
