主效应结论（take away）

  • Update / select 对模板机制是盲的。staging（opstream→code）与 naming（dense→sparse）在 update10th / select / updateStorm / selectStorm 上的因子在各规模都落在 ±10% 噪声带内 — 与 ops 级 factorial（所有 cell 的 update 帧数、native 调用完全相同）互证。模板只改变首帧由谁构建，不改变洞怎么写。
  • Create 的模板收益与子树静态占比成正比。table app 的 create 被动态 v-for 行主导（staging 因子 create +5.5%~+5.6% ≈ 噪声）；静态重内容（sfc-probe FCP 阶梯）上 Code-Template 是唯一全场景为负的因子。要 ET 收益，先看你的首屏静态占比。
  • render 轴（vdom→vapor）才是 update 的大杠杆：updateStorm -54.1% / -54.7%（1k→30k），远超任何模板轴。交互性能选 render model，别指望模板。
  • IFR 是首帧杠杆，不是交互杠杆：ifr 因子只在 create/FCP 显著，update 因子在噪声内；×4 下 vapor 的 IFR 首帧代价由 sparse naming 收回（dense +12% → sparse +2%）。
  • naming（dense→sparse）是内存/簿记轴：BG shells −94%、MT 表 −92%（精确计数），FCP 因子随主机与规模在噪声带内摇摆 — 它是阶梯的必要卫生，不是 FCP 特性。
  • engine 与 engine-et 在本环境记为 N/A（Lynx for Web 无引擎 ET PAPI；__VUE_LYNX_ENGINE_ET_STATUS__ = stub）。它们的解释回退对照样本仅用作 fail-safe 成本与噪声尺，不作为 engine 结论；这条轴已端到端可跑，等引擎 PAPI 落地即测真值。
