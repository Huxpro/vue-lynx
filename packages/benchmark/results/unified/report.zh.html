基线per-node addressing —— 最朴素、最安全：每个节点独立命名，无需元数据、无需校验。vdom = op stream；vapor = named tree。
+bblock templates：用模板块的 parts 信息做块式命名（base+offset）并物化模板。staging 参数缺省为该 render model 的自然档位 —— vdom +b ≡ +b:c（code，baked create()）；vapor +b ≡ +b:d（data：序列化树在运行期作为数据过线——REGISTER_TREE 每模板一次——由 MT 解释；Split 拓扑所致）。注意：两个 render model 的 +b 因子因此不可互比。信息来源也不同：vdom 是 intrinsic（Vue Block 声明），vapor 是 recovered（编译期恢复，带指纹 fail-safe）。
+b:cvapor 的 staging 升到 code 档（#337）：构建期解析 template() HTML，直线 PAPI create() 编进 MT bundle，实例化只过一条 INSTANTIATE_TEMPLATE(id)——residual 运行期完全不过线。命名沿用 base+offset 前序，update 路径零改动；构建期解析与运行期解析的结构指纹不合时静默回退 REGISTER_TREE 数据路径。
+b!delivery 列单独翻转（#338）：staging 仍为 data，但序列化树在构建期烧进 MT bundle（registerVaporStructure(hash, ast)）；BG 只发指纹哈希（REGISTER_TREE_BUNDLE），结构字节 0 过线。解释、命名、update 路径与 vapor +b 逐字节一致；哈希不合时静默回退完整 REGISTER_TREE。
+b:estaging 升到 engine 档：模板常驻引擎（__CreateElementTemplate 家族），native clone。作用于 persistent 本体树。本环境（Lynx for Web）无该 PAPI → N/A。
delivery（第六列性质）residual 何时到 MT：runtime = 运行期过线（vapor +b 的 data 现状与 native）；bundle = 构建期编进 MT bundle（vdom 的 code、RL Snapshot、vapor 的 +b! 与 +b:c）。+b! 是唯一在其他五列全部不动的前提下单独翻转 delivery 的格——纯 delivery 因子的单列读数。

主效应结论（take away）

Update / select 对模板机制是盲的。+b 因子（vdom 与 vapor 两行）在 update10th / select / updateStorm / selectStorm 上各规模都落在 ±10% 噪声带内 — 与 ops 级 factorial（所有 cell 的 update 帧数、native 调用完全相同）互证。模板只改变首帧由谁构建，不改变洞怎么写。
模板的 create 收益只在静态重的屏上出现。「负 = 更快」。table app 的 create 被动态 v-for 行主导，模板化只覆盖那一小块静态骨架，所以 staging 因子在 create 上 +1.5%~-6% ≈ 噪声；换到静态重内容（sfc-probe FCP 阶梯），block/code 模板就成了唯一在所有规模都为负（更快）的因子。规则：先量你首屏的静态占比，再决定要不要上模板。
render 轴（vdom→vapor）才是 update 的大杠杆：updateStorm -65.4% / -58.9% / -49.7%（1k→30k），远超任何模板轴。交互性能选 render model，别指望模板。
IFR 是首帧杠杆，不是交互杠杆：ifr 因子只在 create/FCP 显著，update 因子在噪声内；×4 下 vapor 的 IFR 首帧代价由 +b 收回（基线 +12% → +b +2%）。
那 vapor +b 到底做了什么？在动态表格上它对 create / update / select / storm 延迟几乎没影响（各规模都在 ±10% 噪声内）。它真正可测的收益有两处：(1) 内存 / 簿记 — BG shells −94%、MT 表项 −92%（精确计数）;(2) 静态重首屏 — block 化让首帧构建更省，收益随子树静态占比上升。一句话：+b 是内存 + 静态首帧优化，不是动态延迟杠杆。我们在 +b / +ifr 上花的功夫，产出的是这两样，而不是表格交互延迟。
+b:e 与 +ifr:e 在本环境记为 N/A（Lynx for Web 无引擎 ET PAPI；__VUE_LYNX_ENGINE_ET_STATUS__ = stub）。它们的解释回退对照样本仅用作 fail-safe 成本与噪声尺，不作为 engine 结论；这条轴已端到端可跑，等引擎 PAPI 落地即测真值。
+b!（delivery 单列翻转，#338）与 +b:c（code 档，#337）已实测——收益是「小模板 × 多实例」条件收益，不是普适收益。storms table app（小行模板 × 上万实例）：+b! create -2.5% / +4% / +5.7%、+b:c create -6.5% / +1.3% / +1.6%（1k/10k/30k），update/select 因子在 reps=2 噪声包络内（对照解释路径完全相同的 +b:e stub 行在 30k 同样摆动 ±20%——模板盲性再次验证）；结构字节按设计 0 过线（table app 注册线路 1165 B → 99 B 指纹帧；+b:c 连注册都不发，wire-bytes 计数内嵌在 factor JSON）。但 content 首屏探针（一整张 mega 静态模板 × 单实例）方向相反：×1 下两格 FCP 随规模变慢（最高 ~+16%），机制 = bundle 内驻的 residual 在 MT 启动关键路径上被解析——30k 时 MT 段 gzip 从 8 kB 涨到 +b! 14 kB / +b:c 487 kB（raw 24 kB → 1.1 MB / 3.5 MB）。原猜测 +b:c create −5…−17%（随静态占比）来自 vdom ET「每卡片一个小 create()」的先例，不迁移到 vapor 的单一 mega-template 形态；+b! 的 structure wire −100% 逐字节成立。资格规则：烘焙只在 residual 小且实例多（列表行）时净赚；单实例大模板应留在 data 线路——按模板尺寸门控。update 路径与 vapor +b 逐字节相同（指纹 fail-safe + mutated-hash 测试兜底）。
