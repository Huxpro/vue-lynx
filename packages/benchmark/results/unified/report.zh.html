标记含义
基线per-node addressing —— 最朴素、最安全：每个节点独立命名，无需元数据、无需校验。vdom = op stream；vapor = named tree。
+bblock templates：用模板块的 parts 信息做块式命名（base+offset）并物化模板。staging 参数缺省为该 render model 的自然档位 —— vdom +b ≡ +b:c（code，baked create()）；vapor +b ≡ +b:d（data：序列化树在运行期作为数据过线——REGISTER_TREE 每模板一次——由 MT 解释；Split 拓扑所致）。注意：两个 render model 的 +b 因子因此不可互比。信息来源也不同：vdom 是 intrinsic（Vue Block 声明），vapor 是 recovered（编译期恢复，带指纹 fail-safe）。
+b:estaging 升到 engine 档：模板常驻引擎（__CreateElementTemplate 家族），native clone。作用于 persistent 本体树。本环境（Lynx for Web）无该 PAPI → N/A。+b:c（vapor 的 code 档，M3a）未实现。
delivery（第六列性质，暂无独立 flag）residual 何时到 MT：runtime = 运行期过线（vapor 的 data/native 现状）；bundle = 构建期编进 MT bundle（vdom 的 code、RL Snapshot）。构建期交付 data 的格（候选记法 +b!）尚未实现——可砍掉 BG 解析/序列化/wire/register 四笔 create 期成本。
+ifrIFR：MTS 抢跑 ephemeral 首帧副本，BG 启动后 hydration 采纳或整体重放。paint 参数缺省 = 首帧继承本体 staging。
+ifr:e仅首帧副本用 engine 档画（旧名 engine-et）；本体树照常。区别于 +b:e（全程）。本环境 N/A。同理 +ifr:c（旧名 disposable-et）。

主效应结论（take away）

  • Update / select 对模板机制是盲的。+b 因子（vdom 与 vapor 两行）在 update10th / select / updateStorm / selectStorm 上各规模都落在 ±10% 噪声带内 — 与 ops 级 factorial（所有 cell 的 update 帧数、native 调用完全相同）互证。模板只改变首帧由谁构建，不改变洞怎么写。
  • Create 的模板收益与子树静态占比成正比。table app 的 create 被动态 v-for 行主导（staging 因子 create -15.8%~-1.7% ≈ 噪声）；静态重内容（sfc-probe FCP 阶梯）上 Code-Template 是唯一全场景为负的因子。要 ET 收益，先看你的首屏静态占比。
  • render 轴（vdom→vapor）才是 update 的大杠杆：updateStorm -53.2% / -54.1% / -2.7%（1k→30k），远超任何模板轴。交互性能选 render model，别指望模板。
  • IFR 是首帧杠杆，不是交互杠杆：ifr 因子只在 create/FCP 显著，update 因子在噪声内；×4 下 vapor 的 IFR 首帧代价由 +b 收回（基线 +12% → +b +2%）。
  • vapor 的 +b 是内存/簿记优化：BG shells −94%、MT 表 −92%（精确计数），FCP 因子随主机与规模在噪声带内摇摆 — 它是阶梯的必要卫生，不是 FCP 特性。
  • +b:e 与 +ifr:e 在本环境记为 N/A（Lynx for Web 无引擎 ET PAPI；__VUE_LYNX_ENGINE_ET_STATUS__ = stub）。它们的解释回退对照样本仅用作 fail-safe 成本与噪声尺，不作为 engine 结论；这条轴已端到端可跑，等引擎 PAPI 落地即测真值。
