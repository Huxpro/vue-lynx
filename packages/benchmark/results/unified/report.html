Main-effect takeaways

  • Update and select are template-blind. The staging (opstream→code) and naming (dense→sparse) factors on update10th / select / updateStorm / selectStorm sit inside the ±10% noise band at every scale — corroborating the ops-level factorial (identical update frames + native calls in every cell). Templates change who builds the first frame, not how holes are written.
  • Create benefit scales with the static fraction of the subtree. The table app's create is dominated by dynamic v-for rows (staging factor on create +5.5%…+5.6% ≈ noise); on static-heavy content (the sfc-probe FCP ladder) Code-Template is the one factor negative in every scenario. Check your first screen's static fraction before reaching for ET.
  • The render axis (vdom→vapor) is the update lever: updateStorm -54.1% / -54.7% (1k→30k) — far beyond any template axis. Pick the render model for interaction performance; don't expect templates to move it.
  • IFR is a first-frame lever, not an interaction lever: the ifr factor is significant only on create/FCP, its update factors sit in noise; under ×4 vapor's IFR first-frame cost is recovered by sparse naming (dense +12% → sparse +2%).
  • Naming (dense→sparse) is the memory/bookkeeping axis: −94% BG shells, −92% MT table entries (exact counts) while its FCP factor wobbles inside noise across hosts and scales — necessary ladder hygiene, not an FCP feature.
  • Engine and engine-et are N/A on this host (Lynx for Web has no engine ET PAPI; __VUE_LYNX_ENGINE_ET_STATUS__ = stub). Their interpretation-fallback control samples serve only as a fail-safe-cost / noise yardstick, never as engine conclusions; the axis runs end-to-end and reports real numbers the day the engine PAPI ships.
