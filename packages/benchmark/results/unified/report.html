Conclusions (start here)

Each card: one-line takeaway → so what (how to use it) → key numbers to verify. Never ratio ms across environments.

Ship Vapor for point updates (~8.2× vs VDOM @10k selectStorm)

so what Select = point update (selection / a few classes). High-frequency point interaction is what users feel; one-shot taps often sit on the frame floor. Default Vapor for interaction-heavy lists.

verify VDOM 367 ms → Vapor 45 ms; ~23× vs ReactLynx (1018 ms)

BG micro confirms it: Vapor select ~9.8×

so what Instrumented BG measures the pipeline; storms are the user-visible version. Same direction on both instruments → not a measurement artifact.

verify vdom bg 2.95 ms → vapor 0.30 ms

ReactLynx still wins bulk create (~1.16× vs VDOM @10k)

so what Snapshot bulk-instantiate is great at laying down many nodes once. Creation ≠ update — don’t mix the two stories.

verify ReactLynx 689 · VDOM 798 · Vapor 853 ms

First frame: ReactLynx lowest (FCP@10k 228 ms)

so what On the same-density probe, Snapshot+IFR (RL default) beats every Vue cell. For Vue first-frame, default to IFR+ET.

verify ReactLynx 228 · VDOM+IFR+ET 278 · VDOM 310 ms

Don’t ship IFR without ET: -10% @1k → +22% @10k

so what “−19% FCP” is not a constant. Plain IFR becomes a liability on mid/large trees; ET is the scale hedge. Don’t document “just enable IFR.”

verify VDOM+IFR vs off: 1k -10% · 10k +22%

ET isn’t first-frame-only: post-mount storms ~28% faster

so what Template clone speeds later repeated create/update. Plain IFR ≈ off; ET is what keeps paying off.

verify selectStorm@10k: off 367 · IFR 372 · IFR+ET 264 ms

Scaling shape: nothing exponential; FCP is sublinear

so what Fit cost ∝ N^α: FCP α≈0.7 (sublinear — fixed overhead amortizes); create/update storms ~linear. The separator is Vapor select (stays flatter as N grows).

verify FCP α≈0.68–0.79 · create α≈0.95 · updateStorm α≈1.0–1.2 · React/VDOM selectStorm α≈1.05 (Vapor lower)

Cheat sheet: create→ReactLynx · point/batch updates→Vapor · Vue first-frame→IFR+ET

so what Select = point, Update = batch — don’t mix them. The three old benches fought; on one ladder the product choice collapses to this line.

verify See storm + FCP tables below; never divide ms across those instruments.

Quote ratios, not absolute milliseconds

so what Same protocol, different hosts can move medians 2×+. Portable claims are same-host ratios.

verify Ex: older playground React selectStorm@10k ≈2544 ms; this host ≈1018 ms.

Storm scaling (1k → 30k)

Linear axes, zero baseline — absolute gaps, not log-compressed shape. Select charts = point updates; Update charts = batch throughput.

select storm (point) vs table size

Point updates: wall time for 30 sequential selects — each tick touches selection state only. Vapor stays flat; React climbs; IFR+ET pulls VDOM down.

update storm (batch) vs table size

Batch updates: wall time for 50 multi-row update passes. Creation is separate — sustained batch throughput only.

create vs table size

ReactLynx still leads creation. IFR+ET slightly helps VDOM create via template clone.
