# Playground: React vs Vue on Lynx

Every number in the [VDOM vs Vapor benchmark](/guide/benchmark-vapor) comes
from apps on Lynx for Web — which means you can run the same apps here, on
this page, on your phone. Pick a configuration, tap the same buttons the
harness clicks, and feel the difference. The badge shows **tap → DOM settled**
for your last tap (the last DOM mutation after your tap, once mutations go
quiet).

<BenchPlayground />

**What to try**

1. **Create 10,000 rows**, then **Select storm** (30 selection moves) —
   Vue Vapor ~0.2 s, Vue VDOM ~1 s, React ~3 s.
2. On 10,000 rows, **Update storm** (50 full update passes) —
   Vapor ~1.2 s, VDOM ~2 s, React ~6 s.
3. Single taps (Update / Select on a 1k table) feel identical across
   configurations — frameworks separate under sustained throughput, which is
   what the storms amplify.

## Conclusions

| Finding | Detail |
| --- | --- |
| One-shot ops | Near **parity** at 1k rows; within ~1.5–2× at 10k |
| Storm throughput | ReactLynx ~**3–5×** Vue — same order of magnitude as keyed React vs Vue on the web |
| Where Vue / Vapor lead | Sustained updates; Vapor especially on **select** (two class effects vs O(rows) per tick) |
| Where React leads | **Creation** (~1.2× through 30k) — a ReactLynx adapter trait (snapshot bulk-instantiate), not a framework trait |
| Scale (1k→30k) | Creation linear for everyone; update storms **converge**; select storms stay structurally separated |

We also measured three React optimization levels (manual `memo` +
`useCallback`, naive, and [React Compiler](https://react.dev/learn/react-compiler)).
Compiler lands between naive and manual memo — it memoizes *within* a
component and cannot prune keyed-list row re-renders the way a
component-boundary `memo()` does. Try all three in the playground above.

## How this differs from js-framework-benchmark

Same op set (create/update/select/swap/remove/append/clear on 1k–10k rows,
keyed, idiomatic state) — three differences:

1. **Runtime under test is Lynx's dual-thread pipeline**, not the browser
   DOM. Latency is *click → composed DOM end state* (polled per animation
   frame), not paint tracing. Resolution is one frame (~17 ms), so the table
   ranks multi-frame scenarios.
2. **Sustained-session dimensions** — cold single-shot *and* storms (one tap
   = 50 update passes or 30 selection moves) that lift sub-frame per-tick
   costs above the frame floor.
3. **Framework scope** — ReactLynx, Vue Lynx VDOM, and Vue Lynx Vapor on the
   runtime they actually target.

## Caveats

- Harness is **Lynx for Web in Chromium**, not a native device; Vue 3.6 and
  ReactLynx versions are pinned in the raw results.
- An earlier draft of these numbers showed ReactLynx storms at minutes/DNF.
  That was a Lynx-for-Web profiling shim (`lynx.profile:` never cleared
  measures; ReactLynx called it, Vue did not). The harness now neutralizes
  those entries for every framework — native-parity behavior. Full audit
  trail: `plans/0711-1-cross-framework-benchmark.md`.
