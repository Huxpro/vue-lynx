# Vapor Mode <Badge text="v0.6" type="tip" />

[Vapor mode](https://github.com/vuejs/core/releases) is Vue's
compilation-based rendering mode that ships with Vue 3.6: component templates
compile to code that manipulates elements directly, with no Virtual DOM and
no per-component diffing. Vue Lynx runs Vapor components on Lynx's dual-thread
architecture — with large, measured wins on the update path
([benchmark](/guide/benchmark-vapor)).

:::warning Experimental
Vapor mode itself is **not part of any stable Vue release yet** — as of
mid-2026 it is published in the Vue 3.6 beta / prerelease line only, and the
Vue team labels it unstable. Vue Lynx's Vapor support inherits that status:
try it, but don't bet the app on it. Vue Lynx pins `vue@3.6.0-beta.17`.
:::

## Enabling Vapor mode

Vapor is a **per-app, build-time decision**. Switching an app over touches
three places — the plugin flag, the entry import, and the `vapor` attribute
on each component:

```ts title="lynx.config.ts"
import { defineConfig } from '@lynx-js/rspeedy'
import { pluginVueLynx } from 'vue-lynx/plugin'

export default defineConfig({
  plugins: [
    pluginVueLynx({
      vapor: true, // [!code highlight]
    }),
  ],
})
```

```ts title="src/index.ts"
import { createApp } from 'vue-lynx/vapor' // [!code highlight]
import App from './App.vue'

createApp(App).mount()
```

```vue title="src/Counter.vue"
<script setup vapor lang="ts">
import { ref } from 'vue'

const count = ref(0)
</script>

<template>
  <view class="button" @tap="count++">
    <text>Count: {{ count }}</text>
  </view>
</template>
```

Inside components nothing else changes: `import { ref } from 'vue'` keeps
working because the plugin aliases `'vue'` to `vue-lynx/vapor` when
`vapor: true`.

The entry is `vue-lynx/vapor` (earlier preview names `vue-lynx/with-vapor` and
`vue-lynx/vapor-app` are gone).

## How it works on Lynx

Upstream `@vue/runtime-vapor` drives the browser DOM directly — there is no
custom-renderer hook like the Virtual DOM's `createRenderer()`. Vue Lynx
instead makes the Background Thread's ShadowElement tree *DOM-compatible*:
the compiled Vapor code and its Lynx-specific helper overrides call standard
DOM APIs — `insertBefore`, `cloneNode`, `setAttribute`, `addEventListener`,
`el.style` — and those calls emit the same flat ops stream to the Main
Thread that vdom mode uses.

```
compiled vapor component
  → import from 'vue' (aliased to vue-lynx/vapor)
    → @vue/runtime-vapor            (unmodified upstream)
      → document / Node / Element shims
      → ShadowElement DOM-compat surface
        → ops stream → Main Thread → native elements
```

A few Lynx-specific translations happen along the way:

| Web concept | Lynx translation |
|---|---|
| Template HTML strings (`template("<view class=…>")`) | Parsed by a built-in parser into inert prototypes; cloned per instance |
| Document-level event delegation | Compiled away (`eventDelegation: false`); events register per element |
| `@tap.stop` | Native `catchEvent` (Lynx's stop-propagation) |
| `:bindtap` / `:catchtap` / `:global-bind*` props | Routed to native Lynx event registration (same handling as vdom mode), including fallthrough onto a child component's root |
| `v-model` on `<input>` | Lynx `input` / `confirm` (`.lazy`) events carrying `detail.value` |
| Scoped CSS `data-v-*` attributes | Composable scope **classes** (the build rewrites `[data-v-*]` selectors to class selectors; same mechanism as vdom mode) |

### Template cloning across threads

Vapor's rendering primitive is *clone a template, then bind*. Sending every
cloned element as individual create/append ops would make Vapor's
cross-thread traffic strictly worse than vdom's, so Vue Lynx teaches its own
ops protocol two template instructions:

- `REGISTER_TREE(treeId, structure)` — sent **once per template**,
  carrying the static structure tree.
- `CLONE_TREE(treeId, baseUid)` — sent **once per instance**; both
  threads assign element uids by walking the structure in the same
  pre-order from `baseUid`, so no id mapping needs to be transferred.

These are Vue Lynx's own opcodes (the Main Thread interpreter ships in the
same bundle as the Background Thread code), not a Lynx engine change. The
effect on a 1,000-row list creation: **7,000 ops / 160 KB** across the
thread boundary in Vapor mode vs **17,000 ops / 327 KB** in vdom mode.

## Instant first-frame rendering

Vapor apps can also paint synchronous content during Lynx `loadTemplate`.
Add `enableIFR: true` beside `vapor: true`:

```ts title="lynx.config.ts"
pluginVueLynx({
  vapor: true,
  enableIFR: true,
})
```

Vapor IFR reuses the compiled `template()` / tree protocol — it does **not**
use VDOM Element Templates. Details, trade-offs, and Vapor-specific limits:
[IFR on Vapor](/guide/ifr#ifr-on-vapor).

## Performance

**Bottom line:** update-path operations run **5.8–9.8× faster on the
background thread** (2.1–6.3× end-to-end), creation is at parity, and
cross-thread traffic roughly halves — at the cost of +26% bundle (gzip).

Full methodology and raw data:

- [VDOM vs Vapor benchmark](/guide/benchmark-vapor)
- [Playground: React vs Vue](/guide/benchmark-playground) (includes Vapor)

## API surface

The Vapor entry exports everything a Vapor app imports from `'vue'` — the
standard Vue APIs, SFC macros, Lynx utilities, and the Vapor helper surface —
while vdom-only APIs (`h()`, `<Transition>`, `<Teleport>`, …) are
intentionally absent and fail at build time. Full listing:
[`vue-lynx/vapor` API reference](/guide/api/vapor/).

## Current limitations

- **One app, one mode.** Vue Lynx does not support mixing vdom and Vapor in
  the same app (`vaporInteropPlugin` is not wired up yet — upstream hard-codes
  it to the browser DOM renderer). Mode is a strict either/or, so Vue Lynx
  ships two pure entries instead of one entry pruned by tree-shaking:

  | Entry | Contains | Use for |
  |---|---|---|
  | `vue-lynx` | vdom renderer + Vue runtime-core | Regular (default) apps |
  | `vue-lynx/vapor` | Vapor runtime + Lynx DOM-compat layer | Vapor apps |

  Side effect: vdom-only APIs simply don't exist on `vue-lynx/vapor`, so
  accidentally using them **fails at build time**.

- **No `<Transition>` / `<Teleport>` / `<KeepAlive>` / `<Suspense>`** in
  Vapor apps — upstream Vapor hasn't shipped them yet.
- **No `v-html`** — Lynx elements cannot parse HTML strings (same as vdom-mode
  Vue Lynx).
- **No SSR / HTML hydration** — Lynx has no SSR. (Dual-thread [IFR
  hydration](/guide/ifr#hydration) is a separate mechanism.)
- Vapor-mode Vue constraints apply as usual: `<script setup>` only, no
  Options API, `getCurrentInstance()` returns `null`.
- Vapor IFR requires deterministic, thread-neutral initial rendering.
  Synchronous `setup()` side effects and watchers run in both realms; see
  [IFR limitations](/guide/ifr#account-for-current-limitations).
