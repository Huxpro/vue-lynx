# IFR Benchmarks

This page collects the measurement campaigns behind [Instant First-Frame
Rendering](/guide/ifr) and Element Templates. Three campaigns answer three
different questions, and one methodology lesson connects them: **a
single-process measurement can only see JS cost — IFR's headline win is
structural (removing background-thread boot + IPC from the critical path) and
only appears across a real thread boundary.** Numbers below are medians;
full methodology and raw data live in the repo
([strategy benchmark](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/REPORT.md),
[all-examples sweep](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/EXAMPLES-REPORT.md),
[real-browser verification](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/VERIFICATION.md)).

## 1. Strategy ladder — what each IFR design costs in JS

Before converging on the shipped design, seven rendering strategies were
prototyped and measured against the same logical first screens, through the
same Element PAPI surface. `bg-baseline` and `ifr-replay` are real shipped
code; `ifr-block-tpl` became the shipped Element Templates; the rest are
prototypes kept for reference. All variants produce byte-identical rendered
documents (verified against a real jsdom PAPI environment).

Scenes are generated from one DSL at ~1000–1400 elements: **static-heavy**
(landing/skeleton, 99.7% template-static), **content** (card feed, 37%),
**list** (v-for, 12%).

### Warm render time, `--jitless` V8 (interpreter ≈ main-thread engine proxy)

| variant | static-heavy | content | list |
|---|---|---|---|
| bg-baseline (no IFR) | 11.86 ms | 9.36 ms | 6.87 ms |
| ifr-replay (shipped `enableIFR`) | 10.94 ms | 8.40 ms | 6.57 ms |
| ifr-direct (prototype) | 8.17 ms | 6.36 ms | 4.89 ms |
| ifr-static-tpl (prototype) | 1.11 ms | 5.29 ms | 4.00 ms |
| **ifr-block-tpl (shipped `enableElementTemplates`)** | **0.74 ms** | **1.26 ms** | **1.44 ms** |
| ifr-vapor (prototype upper bound) | 0.53 ms | 0.55 ms | 0.46 ms |
| papi-floor (reference) | 0.54 ms | 0.39 ms | 0.31 ms |

### Cold first run, `--jitless` (models a device's one-shot first frame)

| variant | static-heavy | content | list |
|---|---|---|---|
| ifr-replay | 18.0 ms | 16.4 ms | 11.7 ms |
| **ifr-block-tpl** | **4.2 ms** | **5.3 ms** | **4.9 ms** |
| ifr-vapor | 1.4 ms | 1.2 ms | 0.8 ms |

### Ops payload that would cross the thread boundary

| variant | static-heavy | content | list |
|---|---|---|---|
| replay / baseline | 77.6 KB | 60.4 KB | 45.1 KB |
| **block templates** | **69 B** | **9.2 KB** | **17.1 KB** |

Reading: replay IFR does not reduce JS work — it relocates it (its win is
*when* the frame paints, not how much JS runs). Element templates are the
inflection point: 7–15× on every scene shape, and the protocol shrink also
benefits the normal pipeline and every update. The Vapor-style no-vdom design
sits at the PAPI floor and remains the tracked endgame; its prototype omits
reactive-effect bookkeeping, so treat its numbers as an optimistic bound.

## 2. All examples — FCP, TTI, and size on real bundles

Every example in the repo was built in three configurations (off / ifr /
ifr+et) and executed as real bundle halves in a PAPI-over-jsdom environment
(5 runs each, medians, `--jitless`, device proxy: main-thread parse excluded
as lepus ships precompiled bytecode). 22 of 23 buildable examples render
byte-identical final documents across all three configurations.

**Medians: FCP ×1.04 (ifr) / ×1.02 (ifr+et), TTI ×1.36 / ×1.35,
`main.lynx.bundle` gzip ×2.26.** In pure JS-work terms FCP is neutral — see
campaign 3 for where the real win comes from.

The doubled bundle is IFR's headline cost: the main-thread section grows from
~17 KiB (worklet registrations only) to 78–195 KiB (Vue runtime + app copy).
Element templates add ~1% on top. The sweep also surfaced and fixed three
real bugs (a `SystemInfo` clobber, numeric-style baking bypassing auto-px
normalization, and CSS Modules crashing the main-thread bundle) — the
full per-example tables for all four engine modes are in the
[sweep report](https://github.com/Huxpro/vue-lynx/blob/main/packages/ifr-bench/EXAMPLES-REPORT.md).

## 3. Real threads — Lynx for Web, with ReactLynx as the control

The tables above show FCP roughly *flat* — because in a single process the
JS work is conserved. To verify that the win is structural, the same bundles
were measured on Lynx for Web: the background runtime runs in a genuine Web
Worker with real postMessage IPC, in headless Chromium (FCP = `<lynx-view>`
insertion → first painted content, medians of 7 fresh browser contexts).
ReactLynx serves as the reference control, with its IFR emulated off via an
empty main-thread first screen.

First, the control experiment: **in the same single-process harness,
ReactLynx also shows flat FCP (±4%) and the same TTI regression** — the
reference implementation cannot demonstrate its own headline feature without
a thread boundary either. Then, with real threads:

### No CPU throttle

| bundle | FCP off | FCP ifr | Δ | FCP ifr+et |
|---|---|---|---|---|
| vue hello-world | 92.0 ms | 74.3 ms | **−19%** | 73.4 ms |
| vue gallery (304 nodes) | 127.9 ms | 106.2 ms | **−17%** | 110.6 ms |
| vue hackernews-css | 114.9 ms | 114.1 ms | ~0% | 114.4 ms |
| **ReactLynx probe (85 nodes)** | 99.2 ms | 68.9 ms | **−31%** | — |

### 4× CPU throttle (mobile-class proxy)

| bundle | FCP off | FCP ifr | Δ | FCP ifr+et |
|---|---|---|---|---|
| vue hello-world | 283.4 ms | 266.2 ms | −6% | 268.5 ms |
| vue hello-world (earlier session) | 402.9 ms | 354.4 ms | **−12%** | **333.3 ms (−17%)** |
| vue hackernews-css | 314.0 ms | 377.4 ms | **+20%** ⚠ | 391.9 ms |
| **ReactLynx probe** | 303.1 ms | 266.6 ms | **−12%** | — |

Vue Lynx's IFR captures the same class of benefit as ReactLynx's (−17…−19%
vs −31% on comparable screens), and element templates pull ahead of plain
IFR under CPU throttling. The hackernews **inversion** (+20%) is the profile
guidance made visible: the web platform has no lepus bytecode precompilation,
so the doubled bundle's parse cost lands on the FCP path — a big bundle with
a skeleton-only first screen pays size for no win. On native, precompiled
bytecode shrinks that term toward zero.

### How far is this from plain single-threaded web?

To separate the dual-thread tax from the platform layer, the same screens
were rendered by vanilla single-threaded baselines: plain Vue
(`@vue/runtime-dom`) and plain Preact (ReactLynx's exact preact fork pointed
back at the real DOM) — each **warm** (framework parsed before t0) and
**cold** (framework parse on the FCP path). Decomposition on the ReactLynx
probe screen, no throttle:

| component | cost | evidence |
|---|---|---|
| render 85 elements + paint (preact on DOM) | ~16 ms | plain-preact warm |
| + framework fetch+parse on the FCP path | +13 ms | cold − warm |
| + the Lynx-for-Web platform layer | +40 ms | rl-ifr − plain-preact cold |
| + background boot + hydration IPC round-trip | +30 ms | rl-noifr − rl-ifr |

The thread boundary costs ~30 ms and **IFR removes exactly that slice**. The
remaining gap vs plain web is the web-host emulation layer (worker spawn,
bundle protocol, PAPI→DOM bridging, WASM style engine) — a per-host constant
that native Lynx does not pay, where the platform layer is native code.

## 4. Native engine observations

The full example suite was validated on a native simulator (LynxExplorer,
Lynx SDK 1.4 / PrimJS): 24/25 examples pass with **zero hydration
mismatches**, and off vs IFR+ET screens are visually identical (SSIM ≥
0.9977; scoped CSS, baked inline styles, and auto-px semantics all correct
on the native engine). Single-sample cold-start recordings show first
content ~0.3 s earlier (gallery) up to ~1.2 s earlier (7guis) with IFR —
directional confirmation of the mechanism, not a stable benchmark.

## Reproduce

```bash
# strategy ladder + correctness oracle
pnpm --filter vue-lynx-ifr-bench run check
pnpm --filter vue-lynx-ifr-bench run bench

# all-examples sweep
node packages/ifr-bench/examples-sweep/orchestrate.mjs <dir>
node packages/ifr-bench/examples-sweep/sweep.mjs <dir>

# real-browser (Lynx for Web) measurement, with plain-web baselines
node packages/ifr-bench/web-harness/run-browser.mjs <bundlesDir> [runs] [throttle]
```
