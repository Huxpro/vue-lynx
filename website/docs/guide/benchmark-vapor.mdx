# Benchmark: VDOM vs Vapor

> Product matrices (black-box storms, FCP):
> **[Unified Benchmark Matrix](/guide/benchmark-unified)**.
> This page keeps what that report does **not** show — instrumented BG/e2e,
> cross-thread traffic, and bundle / memory cost for Vue VDOM vs Vapor only.

Same app, both modes, full dual-thread pipeline
([Vue core benchmark](https://github.com/vuejs/core/tree/main/packages-private/benchmark)
ported to Lynx). **Select = point update**; **update every 10th = sparse batch**.

## Instrumented BG / e2e

| Operation | kind | vdom bg | vapor bg | bg | vdom e2e | vapor e2e | e2e |
|---|---|---|---|---|---|---|---|
| select row | point | 2.95 ms | 0.30 ms | **9.8×** | 3.45 ms | 0.55 ms | 6.3× |
| update every 10th | batch | 3.20 ms | 0.55 ms | **5.8×** | 4.85 ms | 2.30 ms | 2.1× |
| swap rows | point-ish | 3.90 ms | 0.65 ms | **6.0×** | 4.65 ms | 1.20 ms | 3.9× |
| remove row | point | 2.85 ms | 0.40 ms | **7.1×** | 3.40 ms | 0.90 ms | 3.8× |

e2e &lt; bg because both modes emit near-identical ops — the delta is Vue work
on the background thread (shared with your app logic).

Creation e2e is ~parity (vapor create1k ~0.87× … create10k ~0.94×). See
[vapor mode](/guide/vapor-mode#how-it-works-on-lynx) for `REGISTER_TREE` /
`CLONE_TREE`.

## Cross-thread traffic (create 1k)

| | vdom | vapor |
|---|---|---|
| ops per flush | 17,000 | **7,000 (−59%)** |
| JSON bytes | 327 KB | **160 KB (−51%)** |

On Lynx for Web this barely moves e2e (DOM construction dominates). On native
the payload crosses a serialization boundary — halving it matters more there.

## Startup, memory, bundle

| Metric | vdom | vapor | note |
|---|---|---|---|
| First screen (attach → content) | 121.7 ms | 125.8 ms | +3%, CIs overlap |
| Bundle gzip (`main.lynx.bundle`) | 39.2 KB | 49.3 KB | +26% |
| JS heap after 10k rows | 76–136 MB | 76–105 MB | indicative only |
