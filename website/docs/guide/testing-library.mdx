# VueLynx Testing Library

Vue Lynx provides a testing library (`vue-lynx-testing-library`) that lets you test Vue Lynx components in Node.js using [Vitest](https://vitest.dev) — no device or emulator required. It renders components through the full dual-thread pipeline and outputs to JSDOM, so you can query and assert against real rendered output.

If you've used [Vue Test Utils](https://test-utils.vuejs.org/) or [React Testing Library](https://testing-library.com/docs/react-testing-library/intro), the API will feel familiar.

## How It Works

The testing library simulates Lynx's dual-thread architecture in a single Node.js process using [`@lynx-js/testing-environment`](https://www.npmjs.com/package/@lynx-js/testing-environment):

```
┌─────────────────────────┐           ┌─────────────────────────┐
│   Background Thread     │   ops     │     Main Thread         │
│   Vue 3 + Renderer      │ ────────▶ │   PAPI → JSDOM          │
│   (component logic)     │           │   (rendered elements)   │
└─────────────────────────┘           └─────────────────────────┘
```

1. `render()` mounts your Vue component on the background thread.
2. Vue's renderer produces **ops** (operations) that describe element creation, attribute changes, etc.
3. Ops are applied synchronously to the main thread's PAPI layer, which writes to JSDOM.
4. You query and assert against the JSDOM container using `@testing-library/dom` queries.

This means your tests exercise the same code path as production — from Vue reactivity through the ops protocol to native element creation.

### Vitest Configuration

Create a `vitest.config.ts` that aliases `vue-lynx` subpaths to their source (or dist) files and registers the setup file:

```ts title="vitest.config.ts"
import { defineConfig } from 'vitest/config';
import path from 'node:path';

export default defineConfig({
  test: {
    environment: 'jsdom',
    globals: true,
    setupFiles: [path.resolve(__dirname, 'test/setup.ts')],
    include: ['test/**/*.test.ts'],
    alias: [
      {
        find: 'vue-lynx/entry-background',
        replacement: path.resolve(
          __dirname,
          'node_modules/vue-lynx/runtime/dist/entry-background.js',
        ),
      },
      {
        find: 'vue-lynx/main-thread',
        replacement: path.resolve(
          __dirname,
          'node_modules/vue-lynx/main-thread/dist/entry-main.js',
        ),
      },
      {
        find: 'vue-lynx/internal/ops',
        replacement: path.resolve(
          __dirname,
          'node_modules/vue-lynx/internal/dist/ops.js',
        ),
      },
      {
        find: /^vue-lynx$/,
        replacement: path.resolve(
          __dirname,
          'node_modules/vue-lynx/runtime/dist/index.js',
        ),
      },
    ],
  },
});
```

### Setup File

Create a setup file that initializes the dual-thread testing environment. This runs before any test module is imported, so all Lynx globals are in place when Vue's runtime loads.

```ts title="test/setup.ts"
import { JSDOM } from 'jsdom';
import { LynxTestingEnv } from '@lynx-js/testing-environment';

// Create the testing environment
const jsdom = new JSDOM('<!DOCTYPE html><html><body></body></html>');
const lynxTestingEnv = new LynxTestingEnv(jsdom);
(globalThis as any).lynxTestingEnv = lynxTestingEnv;

// Wire Main Thread globals
lynxTestingEnv.switchToMainThread();
if (typeof (globalThis as any).registerWorkletInternal === 'undefined') {
  (globalThis as any).registerWorkletInternal = () => {};
}
await import('vue-lynx/main-thread');

const mainThreadFns = {
  renderPage: (globalThis as any).renderPage,
  vuePatchUpdate: (globalThis as any).vuePatchUpdate,
  processData: (globalThis as any).processData,
  updatePage: (globalThis as any).updatePage,
  updateGlobalProps: (globalThis as any).updateGlobalProps,
};

const mtGlobal = lynxTestingEnv.mainThread.globalThis as any;
Object.assign(mtGlobal, mainThreadFns);

// Wire Background Thread globals
lynxTestingEnv.switchToBackgroundThread();
await import('vue-lynx/entry-background');

const publishEventFn = (globalThis as any).publishEvent;
const bgGlobal = lynxTestingEnv.backgroundThread.globalThis as any;
bgGlobal.publishEvent = publishEventFn;

// Re-wire globals after env resets between tests
(globalThis as any).onSwitchedToMainThread = () => {
  Object.assign(globalThis, mainThreadFns);
};

(globalThis as any).onSwitchedToBackgroundThread = () => {
  if ((globalThis as any).lynxCoreInject?.tt) {
    (globalThis as any).lynxCoreInject.tt.publishEvent = publishEventFn;
  }
  (globalThis as any).publishEvent = publishEventFn;
};
```

## Writing Tests

### Basic Rendering

Use `render()` to mount a component and query the output:

```ts title="test/hello.test.ts"
import { describe, it, expect } from 'vitest';
import { h, defineComponent } from 'vue-lynx';
import { render } from 'vue-lynx-testing-library';

describe('HelloWorld', () => {
  it('renders a greeting', () => {
    const Comp = defineComponent({
      render() {
        return h('view', null, [
          h('text', null, 'Hello Lynx'),
        ]);
      },
    });

    const { container } = render(Comp);
    const textEl = container.querySelector('text');
    expect(textEl).not.toBeNull();
    expect(textEl!.textContent).toBe('Hello Lynx');
  });
});
```

`render()` returns a `RenderResult` containing:

| Property | Description |
|---|---|
| `container` | The JSDOM root element with all rendered children |
| `unmount()` | Unmount the current Vue app |
| `rerender(component, props?)` | Re-render with a new root component |
| `getByText()`, `queryByRole()`, ... | All [`@testing-library/dom`](https://testing-library.com/docs/queries/about) queries, bound to the container |

### Testing Reactivity

After changing reactive state outside of an event handler, call `await nextTick()` twice (once for Vue's scheduler, once for the ops flush to main thread) — or use `waitForUpdate()`:

```ts
import { describe, it, expect } from 'vitest';
import { h, defineComponent, ref, nextTick } from 'vue-lynx';
import { render, waitForUpdate } from 'vue-lynx-testing-library';

describe('reactivity', () => {
  it('updates text when ref changes', async () => {
    const count = ref(0);

    const Comp = defineComponent({
      setup() {
        return () => h('text', null, `Count: ${count.value}`);
      },
    });

    const { container } = render(Comp);
    expect(container.querySelector('text')!.textContent).toBe('Count: 0');

    count.value = 42;
    await waitForUpdate();

    expect(container.querySelector('text')!.textContent).toBe('Count: 42');
  });
});
```

:::tip Why two `nextTick()` calls?
Vue Lynx has a dual-thread architecture. The first `nextTick()` waits for Vue's reactive scheduler to flush on the background thread. The second waits for the ops to be applied on the main thread. `waitForUpdate()` wraps both calls for convenience.
:::

### Testing Events

Use `fireEvent` to simulate Lynx events. Event handlers run through the same pipeline as production — dispatched on the JSDOM element, routed through `publishEvent` to the background thread handler.

```ts
import { describe, it, expect } from 'vitest';
import { h, defineComponent, ref, nextTick } from 'vue-lynx';
import { render, fireEvent } from 'vue-lynx-testing-library';

describe('events', () => {
  it('handles tap events', async () => {
    const count = ref(0);

    const Comp = defineComponent({
      setup() {
        return () =>
          h('view', { bindtap: () => count.value++ }, [
            h('text', null, `${count.value}`),
          ]);
      },
    });

    const { container } = render(Comp);
    expect(container.querySelector('text')!.textContent).toBe('0');

    fireEvent.tap(container.querySelector('view')!);
    await nextTick();
    await nextTick();

    expect(container.querySelector('text')!.textContent).toBe('1');
  });
});
```

#### Supported Events

`fireEvent` provides named helpers for all standard Lynx events:

| Helper | Lynx Event |
|---|---|
| `fireEvent.tap(el)` | `bindEvent:tap` |
| `fireEvent.longtap(el)` | `bindEvent:longtap` |
| `fireEvent.touchstart(el)` | `bindEvent:touchstart` |
| `fireEvent.touchmove(el)` | `bindEvent:touchmove` |
| `fireEvent.touchend(el)` | `bindEvent:touchend` |
| `fireEvent.touchcancel(el)` | `bindEvent:touchcancel` |
| `fireEvent.scroll(el)` | `bindEvent:scroll` |
| `fireEvent.scrollend(el)` | `bindEvent:scrollend` |
| `fireEvent.focus(el)` | `bindEvent:focus` |
| `fireEvent.blur(el)` | `bindEvent:blur` |
| `fireEvent.layoutchange(el)` | `bindEvent:layoutchange` |
| `fireEvent.transitionend(el)` | `bindEvent:transitionend` |
| `fireEvent.animationend(el)` | `bindEvent:animationend` |
| `fireEvent.longpress(el)` | `bindEvent:longpress` |

You can also pass custom event init data:

```ts
fireEvent.tap(el, { detail: { x: 100, y: 200 } });
```

To fire `catchEvent` (non-propagating) handlers instead of `bindEvent`:

```ts
fireEvent.tap(el, { eventType: 'catchEvent' });
```

### Testing Conditional Rendering

```ts
import { describe, it, expect } from 'vitest';
import { h, defineComponent, ref, nextTick, createCommentVNode } from 'vue-lynx';
import { render } from 'vue-lynx-testing-library';

describe('v-if', () => {
  it('toggles content reactively', async () => {
    const show = ref(true);

    const Comp = defineComponent({
      setup() {
        return () =>
          h('view', null, [
            show.value
              ? h('text', null, 'visible')
              : createCommentVNode('v-if'),
          ]);
      },
    });

    const { container } = render(Comp);
    expect(container.querySelectorAll('text').length).toBe(1);

    show.value = false;
    await nextTick();
    await nextTick();

    expect(container.querySelectorAll('text').length).toBe(0);
  });
});
```

### Testing List Rendering

```ts
import { describe, it, expect } from 'vitest';
import { h, defineComponent, ref, nextTick } from 'vue-lynx';
import { render } from 'vue-lynx-testing-library';

describe('v-for', () => {
  it('updates list reactively', async () => {
    const items = ref(['Apple', 'Banana']);

    const Comp = defineComponent({
      setup() {
        return () =>
          h(
            'view',
            null,
            items.value.map((item) => h('text', { key: item }, item)),
          );
      },
    });

    const { container } = render(Comp);
    expect(container.querySelectorAll('text').length).toBe(2);

    items.value = ['Apple', 'Banana', 'Cherry'];
    await nextTick();
    await nextTick();

    const texts = container.querySelectorAll('text');
    expect(texts.length).toBe(3);
    expect(texts[2]!.textContent).toBe('Cherry');
  });
});
```

### Testing with Props

Pass props to the root component via the second argument to `render()`:

```ts
const Greeting = defineComponent({
  props: {
    name: { type: String, required: true },
  },
  render() {
    return h('text', null, `Hello, ${this.name}!`);
  },
});

const { container } = render(Greeting, { name: 'World' });
expect(container.querySelector('text')!.textContent).toBe('Hello, World!');
```

### Re-rendering

Use `rerender()` to replace the root component:

```ts
const { container, rerender } = render(CompA);
expect(container.querySelector('text')!.textContent).toBe('A');

rerender(CompB);
expect(container.querySelector('text')!.textContent).toBe('B');
```

## API Reference

### `render(component, props?)`

Mounts a Vue component through the full dual-thread pipeline and returns a `RenderResult`.

- **`component`** — A Vue component (created with `defineComponent`, `h()`, or an SFC).
- **`props`** *(optional)* — Root props passed to the component.

### `cleanup()`

Unmounts the current Vue app and clears the JSDOM container. Called automatically after each test via `afterEach` — you typically don't need to call this manually.

### `waitForUpdate()`

Waits for all pending Vue reactive updates to flush through the dual-thread pipeline. Equivalent to calling `await nextTick(); await nextTick();`.

### `fireEvent`

Dispatches Lynx events on JSDOM elements. See [Supported Events](#supported-events) for the full list.

```ts
fireEvent.tap(element);
fireEvent.tap(element, eventInit);
fireEvent(element, new Event('bindEvent:tap'));
```

### `screen`

A convenience object from `@testing-library/dom` for querying the rendered output. Works the same as in React Testing Library:

```ts
import { screen } from 'vue-lynx-testing-library';

render(Comp);
expect(screen.getByText('Hello')).not.toBeNull();
```

## Tips

- **Auto-cleanup**: The testing library automatically calls `cleanup()` after each test. You don't need to unmount manually.
- **Imports**: Import Vue APIs (`ref`, `defineComponent`, `nextTick`, etc.) from `vue-lynx`, not from `vue`. This ensures you use the Vue Lynx custom renderer.
- **Lynx elements**: Use `<view>`, `<text>`, `<image>`, etc. — not HTML tags. Query with `container.querySelector('view')`, not `container.querySelector('div')`.
- **`@testing-library/jest-dom`**: You can use matchers like `toHaveTextContent()` for more expressive assertions. Add it to your setup file:
  ```ts
  import '@testing-library/jest-dom';
  ```
