# Playground：React vs Vue on Lynx

[VDOM vs Vapor 基准测试](/zh/guide/benchmark-vapor)里的每个数字都来自跑在
Lynx for Web 上的应用——同一批应用可以直接嵌在这个页面里，在你的手机上运行。
选择一个配置，亲手点按 harness 所点的按钮，自己感受差异。徽标显示最近一次
点按的 **tap → DOM settled** 耗时（点按后最后一次 DOM 变更、以变更静默为准）。

<BenchPlayground />

**建议试玩**

1. 先 **Create 10,000 rows**，再点 **Select storm**（30 次选中移动）——
   Vue Vapor 约 0.2 秒，Vue VDOM 约 1 秒，React 约 3 秒。
2. 在 10,000 行上点 **Update storm**（50 轮全表更新）——
   Vapor 约 1.2 秒，VDOM 约 2 秒，React 约 6 秒。
3. 单次点按（1k 表上的 Update / Select）各配置手感一致——框架只在持续吞吐下
   才拉开差距，这正是风暴所放大的。

## 结论

| 发现 | 说明 |
| --- | --- |
| 单次操作 | 1k 行近乎**持平**；10k 行约 1.5–2× |
| 风暴吞吐 | ReactLynx 约为 Vue 的 **3–5×**——与 Web 上 keyed React vs Vue 同量级 |
| Vue / Vapor 领先处 | 持续更新；Vapor 在 **select** 上尤其明显（两次 class effect vs 每 tick O(行数)） |
| React 领先处 | **创建**（直到 30k 约 1.2×）——ReactLynx 适配层特征（snapshot 批量实例化），不是框架特征 |
| 规模（1k→30k） | 创建对所有人线性；update storm **收敛**；select storm 保持结构性分离 |

我们也测了三档 React 优化（手工 `memo` + `useCallback`、朴素、
[React Compiler](https://react.dev/learn/react-compiler)）。Compiler 落在
朴素与手工 memo 之间——它做的是*组件内部* memo，无法像组件边界的 `memo()`
那样剪掉 keyed 列表的逐行重渲染。三档都可以在上面的 playground 里试。

## 与 js-framework-benchmark 有何不同

同一套操作集（对 1k–10k 行 create/update/select/swap/remove/append/clear，
keyed，各框架惯用状态）——三点不同：

1. **被测运行时是 Lynx 双线程管线**，不是浏览器 DOM。延迟是*点击 → composed
   DOM 终态*（逐动画帧轮询），不是 paint tracing。分辨率约一帧（~17 ms），
   表格只对跨多帧场景排名。
2. **持续会话维度**——冷启动单发，以及风暴（一次点按 = 50 轮更新或 30 次选中），
   把亚帧级每 tick 成本放大到帧地板之上。
3. **框架范围**——ReactLynx、Vue Lynx VDOM、Vue Lynx Vapor，在它们真正面向的
   运行时上。

## 注意事项

- Harness 是 **Chromium 中的 Lynx for Web**，不是真机；Vue 3.6 与 ReactLynx
  版本见原始结果。
- 本页早期草稿曾把 ReactLynx 风暴报成数分钟/DNF。那是 Lynx for Web 的
  profiling shim（`lynx.profile:` 从不清除 measure；ReactLynx 会调用，Vue
  不会）。harness 现已对所有框架中和这些条目。完整审计见
  `plans/0711-1-cross-framework-benchmark.md`。
