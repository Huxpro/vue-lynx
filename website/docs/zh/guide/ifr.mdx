# 首屏直出（IFR） <Badge text="v0.5" type="tip" />

首屏直出（Instant First-Frame Rendering，IFR）会在 Lynx `loadTemplate` 期间、后台线程启动之前绘制真实内容，去掉原先等待后台启动、首次 Vue 渲染和 IPC 造成的白屏。

IFR 在两种渲染器上是同一产品能力；差别在于主线程首帧怎么建出来：

| 渲染器 | 如何开启 | 首帧机制 |
| --- | --- | --- |
| **VDOM** | `enableIFR: true` | 主线程跑 Vue VDOM 渲染，并用[元素模板（Element Templates）](#element-templates)压低同步创建成本 |
| **Vapor** | `vapor: true` + `enableIFR: true` | 走 Vapor 的 `template()` clone（`REGISTER_TREE` / `CLONE_TREE`）——**不用**元素模板 |

## IFR 为什么能消除白屏

没有 IFR 时，可见内容必须等后台线程：

```text
主线程：   空页面 ───────────────────────▶ 应用 ops ─▶ 绘制
后台线程：          启动 ─▶ 渲染 ─▶ IPC ─┘
```

开启 IFR 后，主线程 bundle 携带 Vue 运行时和应用。Vue 在 `loadTemplate` 内同步渲染，后台线程并行启动：

```text
主线程：   loadTemplate ─▶ 渲染 ─▶ 绘制
后台线程：               启动 ─▶ 渲染 ─▶ hydration
```

首屏可以在任何后台 JavaScript 运行之前出现。Hydration 完成后，树由后台线程接管，后续更新全部走正常 ops 管线。

## Hydration

两个线程使用相同初始数据执行同一个应用。主线程记录自己应用的 ops；后台线程的初始批次与该记录协调：

- 结构完全一致的帧会被跳过——内容已在屏上。
- 文本、样式和属性差异会原地 patch。
- 结构差异会拆除首屏树，再按后台 ops 重建。

正确性不依赖两次渲染一致。结构不一致只会失去该屏的 IFR 性能收益，并在开发环境输出 `[vue-lynx] IFR hydration mismatch`。

事件 sign 与元素 ID 在两次渲染中保持确定性。后台 handler registry 就绪后，首帧元素的事件直接路由到正常 Vue handler，无需重新绑定。

## 编写 IFR 友好的首屏

初始渲染会在每个线程各执行一次，应表现为输入的纯函数：

- 保持首屏输出确定性。渲染路径中避免 `Math.random()`、`Date.now()` 和依赖线程的分支。
- 把数据请求、定时器、订阅和其他副作用放进组合式 API 生命周期钩子。这些钩子在主线程渲染期间被抑制，只在后台线程运行。
- Options API 的 `mounted()` 目前不会在主线程被抑制。使用 IFR 的首屏组件应优先用组合式 API。
- IFR 只能绘制同步可用的数据。fetch 驱动页面若需要更早首帧，应提供有意义的外壳或骨架。
- 后台 registry 就绪前的短暂窗口中，普通事件可能被丢弃。[主线程脚本](/zh/guide/main-thread-script) handler 本来就在主线程，因此仍可交互。

## 取舍

- Vue 运行时和应用会打进两个线程的 bundle。全示例中 `main.lynx.bundle` gzip 中位增长约 **2.26×**。
- 应用会在两个线程各求值一次。全示例串行工作量的 TTI proxy 约 **+35–36%**；真机上主线程绘制与后台启动重叠。
- IFR 无法加速只有异步请求完成后才存在的内容。
- CSS Modules 的 class 名可能在首帧尚未哈希，并在 hydration 时 patch 为最终名称。

## VDOM 上的 IFR

在 `lynx.config.ts` 中开启。典型组件树不需要改应用代码：

```ts title="lynx.config.ts"
import { defineConfig } from '@lynx-js/rspeedy'
import { pluginVueLynx } from 'vue-lynx/plugin'

export default defineConfig({
  plugins: [
    pluginVueLynx({
      enableIFR: true,
    }),
  ],
})
```

在 VDOM 入口上，`enableIFR: true` 默认同时开启**元素模板（Element Templates）**。IFR 决定首帧*何时*绘制，元素模板决定静态骨架*创建得多便宜*——这是推荐的 VDOM 路径。

<span id="element-templates" />

### 元素模板（Element Templates）

元素模板是 **仅属于 VDOM** 的编译期优化，不适用于 Vapor。

普通路径中，每个静态元素仍要创建 vnode 和 shadow node、产生多条 ops、跨线程传输，再由主线程解释器分发。编译器可以把符合条件的子树折叠为：

```text
没有 ET：vnode → ShadowElement → 逐节点 ops → 解释器 → PAPI
使用 ET：一个 vnode → INSTANTIATE_TEMPLATE → 直线 PAPI create()
```

静态骨架会变成编译器生成的 JavaScript `create()` 函数。Vue 为整棵子树只发送一条 `INSTANTIATE_TEMPLATE` op；动态文本、class、样式和属性作为“洞”，继续通过普通 `SET_*` ops 更新。

#### 哪些结构会下沉

当子树结构在编译期完全可知时，它可以下沉：每个节点都是普通 Lynx 元素，只有属性值或文本内容是动态的。

```vue
<view class="card">                       <!-- 一棵下沉子树 -->
  <image class="icon" src="a.png" />     <!-- 烘焙静态骨架 -->
  <text class="title">{{ title }}</text> <!-- 动态文本洞   -->
  <view :class="badgeClass">...</view>   <!-- 动态 class 洞 -->
</view>
```

以下结构保留在普通 vnode 路径：

- 组件、slot、`v-if` / `v-for` 宿主（内部的普通元素仍可下沉）
- 内部节点上的运行时指令、ref、key、ID 和 vnode hook
- 有专用主线程行为的 `<list>` 和 `<list-item>`
- 注释和混合动态文本段

模板根节点保留正常 vnode props 和指令。Scoped CSS 通过组件的 scope class token 支持。只有确认与普通样式路径语义一致时，静态 inline style 才会被烘焙。

#### 框架模板，不是引擎二进制模板

Vue Lynx 当前不使用 Lynx 的二进制 `elementTemplates` bundle section、`__ElementFromBinary` 或 `__GetTemplateParts`。这里的元素模板是框架层优化，由普通 typed Element PAPI 组成——去掉 vnode、逐节点 ops、序列化和解释器分发，同时保留现有 renderer 与 hydration 协议。

#### 语义保证

下沉是优化，不是新的渲染模式。不符合条件的结构会自动回退；下沉与未下沉的树必须产生相同的文档与更新。需要身份（`ref`、ID 等）的节点不会进入匿名内部路径。

### VDOM 配置

| 配置 | `enableIFR` | `enableElementTemplates` | 用途 |
| --- | ---: | ---: | --- |
| No IFR | `false` | `false` | 默认的后台线程渲染 |
| **IFR + ET** | `true` | 省略或 `true` | 推荐的 VDOM IFR 路径 |
| IFR without ET | `true` | `false` | 兼容、调试、测量对照的退出开关 |

保留 IFR、二分元素模板问题时：

```ts title="lynx.config.ts"
pluginVueLynx({
  enableIFR: true,
  enableElementTemplates: false,
})
```

也可以在普通后台渲染中单独开启元素模板（`enableElementTemplates: true` 且不开 IFR）。这是高级组合，不是推荐的 IFR 配置。

## Vapor 上的 IFR

Vapor 已将静态结构编译为 `template()` clone。开启 IFR 后，Vue Lynx 将其映射为带确定性元素 ID 的 `REGISTER_TREE` / `CLONE_TREE`，以便主线程在 `loadTemplate` 期间绘制。

Vapor IFR **不使用**元素模板。编译器已经产出可 clone 的树时，VDOM 专属的 ET 变换没有必要。

```ts title="lynx.config.ts" {7-8}
import { defineConfig } from '@lynx-js/rspeedy'
import { pluginVueLynx } from 'vue-lynx/plugin'

export default defineConfig({
  plugins: [
    pluginVueLynx({
      vapor: true,
      enableIFR: true,
    }),
  ],
})
```

`enableIFR` 控制首帧*何时*绘制；`vapor` 选择无 Virtual DOM 的运行时。

:::warning Experimental
Vapor 模式使用 Vue 3.6 预发布包。Vapor IFR 继承该实验状态。上线前请自行测量包体积与首屏。
:::

<span id="account-for-current-limitations" />

### Vapor IFR 限制

在 [Vapor 模式](/zh/guide/vapor-mode#current-limitations)之上：

- 后台 handler 注册完成前发生的普通事件会被丢弃。[主线程脚本](/zh/guide/main-thread-script) 处理器仍可用。
- `useVaporCssVars()` 在后台线程安装响应式 CSS 变量 effect。CSS `v-bind()` 可能在 hydration 时才出现，而不是主线程首帧。
- `onErrorCaptured` 仍然有效，但不会把顶层模块求值包进 fallback 边界。
- 交接后主线程响应式图仍占用内存；其后可见操作会被丢弃。
- 异步首屏数据仍走普通后台更新路径。

CSS Modules 在主线程构建中保留生成的 class map。Scoped CSS 与普通抽取样式与非 IFR 的 Vapor 构建使用同一份样式表输出。

## Benchmark

**结论先行**

| 问题 | 答案 |
| --- | --- |
| IFR 能改善首屏吗？ | **能。** 在 Lynx for Web（真实 Worker 边界）上，十个 example 的 FCP 中位约 **−19%**（默认 IFR），与 ReactLynx 在同一 harness 上的 −23% 同级。 |
| VDOM 要不要跟 ET 一起开？ | **要。** ET 负责随着屏幕变大压住主线程同步渲染成本。没有 ET，IFR 在约 10k 元素时就会慢过纯 VDOM，到 30k 大约 **+20%**；有 ET 则全程最快。 |
| Vapor 呢？ | 同源矩阵上 1× CPU 约 **−9%**，幅度小于 VDOM+IFR+ET。请在目标设备实测——4× CPU 时 MT bundle 成本可能吃掉收益。 |
| 什么时候不要开？ | 响应回来前无物可画的 fetch 驱动首屏，或 gzip 约翻倍不可接受、又未实测的体积敏感场景。 |

### 建议

| 你的首屏 | 建议 |
| --- | --- |
| 内容优先、同步数据 | **开 `enableIFR: true`**（VDOM 自带 ET；Vapor 走树协议） |
| 数据驱动但初始数据同步可得 | 开 |
| fetch 驱动——响应前无物可画 | 不开；或先做出真正的骨架屏再开 |
| 体积极度敏感 | 先实测——gzip 大约翻倍 |

无论哪种画像：保持首屏确定性、副作用放进组合式 API 生命周期钩子（见[编写 IFR 友好的首屏](#编写-ifr-友好的首屏)）。`enableElementTemplates: false` 只用于二分疑似 ET 问题——不是性能调节项。

### 支撑数据

Hello-world FCP（Lynx for Web）、约 1k 元素场景的 warm 渲染成本（V8 `--jitless`），以及全示例 TTI / gzip 相对 No IFR 的中位：

| 配置 | FCP | 渲染成本 | TTI proxy | bundle gzip |
| --- | ---: | ---: | ---: | ---: |
| No IFR | 96.6 ms | 9.4 ms | 1.00× | 1.00× |
| **IFR + ET**（VDOM 默认） | **75.4 ms（−22%）** | **1.3 ms** | 1.35× | ~2.26× |
| IFR without ET | 75.8 ms（−22%） | 8.4 ms | 1.36× | ~2.26× |

同源内容流（约 1k 元素）、五种架构、CPU ×1 / ×4：

| 条目 | 1× CPU | 4× CPU |
| --- | --- | --- |
| VDOM off → IFR（无 ET） | 95 → 77 ms (**−19%**) | 270 → 287 ms (**+7%**) |
| VDOM off → IFR + ET | 95 → 72 ms (**−24%**) | 270 → 254 ms (**−6%**) |
| Vapor off → IFR | 97 → 89 ms (**−9%**) | 304 → 318 ms (**+5%**) |
