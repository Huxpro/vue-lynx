# Vapor Mode <Badge text="v0.6" type="tip" />

[Vapor mode](https://github.com/vuejs/core/releases) 是 Vue 3.6 引入的编译式
渲染模式：组件模板直接编译为操作元素的代码，没有 Virtual DOM，也没有逐组件
diff。Vue Lynx 在 Lynx 双线程架构上运行 Vapor 组件——更新路径有可量化的大幅
性能优势（见[基准测试](/zh/guide/benchmark-vapor)）。

:::warning 实验性
Vapor mode 本身**尚未进入任何 Vue 稳定版本**——截至 2026 年年中，它只发布在
Vue 3.6 beta / prerelease 线中，Vue 团队也标记其为不稳定。Vue Lynx 的 Vapor
支持继承这一状态：欢迎尝试，但别把生产应用押在上面。Vue Lynx 固定使用
`vue@3.6.0-beta.17`。
:::

## 开启 Vapor mode

Vapor 是**以应用为粒度的构建期决策**。切换一个应用需要改三处：插件开关、
入口 import、以及每个组件上的 `vapor` 属性：

```ts title="lynx.config.ts"
import { defineConfig } from '@lynx-js/rspeedy'
import { pluginVueLynx } from 'vue-lynx/plugin'

export default defineConfig({
  plugins: [
    pluginVueLynx({
      vapor: true, // [!code highlight]
    }),
  ],
})
```

```ts title="src/index.ts"
import { createApp } from 'vue-lynx/vapor' // [!code highlight]
import App from './App.vue'

createApp(App).mount()
```

```vue title="src/Counter.vue"
<script setup vapor lang="ts">
import { ref } from 'vue'

const count = ref(0)
</script>

<template>
  <view class="button" @tap="count++">
    <text>Count: {{ count }}</text>
  </view>
</template>
```

组件内部无需其他改动：`import { ref } from 'vue'` 照常工作——`vapor: true`
时插件会把 `'vue'` alias 到 `vue-lynx/vapor`。

入口就是 `vue-lynx/vapor`（早期预览名 `vue-lynx/with-vapor`、
`vue-lynx/vapor-app` 已移除）。

## 在 Lynx 上的工作原理

上游 `@vue/runtime-vapor` 直接驱动浏览器 DOM——它没有像 Virtual DOM 的
`createRenderer()` 那样的 custom renderer 钩子。Vue Lynx 的做法是让后台线程
的 ShadowElement 树*足够 DOM 兼容*：编译产物及 Lynx 专用 helper 覆盖调用
标准 DOM API——`insertBefore`、`cloneNode`、`setAttribute`、
`addEventListener`、`el.style`——这些调用最终发出与 vdom 模式相同的扁平 ops
流到主线程。

```
编译后的 vapor 组件
  → import from 'vue'（alias 到 vue-lynx/vapor）
    → @vue/runtime-vapor            （上游原样复用）
      → document / Node / Element 全局 shim
      → ShadowElement 的 DOM 兼容面
        → ops 流 → 主线程 → 原生元素
```

过程中有几处 Lynx 特定的转译：

| Web 概念 | Lynx 转译 |
|---|---|
| 模板 HTML 字符串（`template("<view class=…>")`） | 内置解析器解析为惰性原型，按实例克隆 |
| 文档级事件委托 | 编译期关闭（`eventDelegation: false`），事件逐元素注册 |
| `@tap.stop` | 原生 `catchEvent`（Lynx 的阻止冒泡机制） |
| `:bindtap` / `:catchtap` / `:global-bind*` 属性 | 路由到原生 Lynx 事件注册（与 vdom 模式同一处理），包括透传到子组件根元素的场景 |
| `<input>` 上的 `v-model` | Lynx `input` / `confirm`（`.lazy`）事件，取 `detail.value` |
| Scoped CSS 的 `data-v-*` 属性 | 可组合的 scope **class**（构建期把 `[data-v-*]` 选择器改写为 class 选择器；与 vdom 模式同一机制） |

### 跨线程的模板克隆

Vapor 的渲染原语是*克隆模板再绑定*。如果把每个克隆出的元素都拆成独立的
create/append op 发送，Vapor 的跨线程流量会比 vdom 更差。因此 Vue Lynx 给
自己的 ops 协议增加了两条模板指令：

- `REGISTER_TREE(treeId, structure)` —— **每模板发送一次**，携带
  静态结构树。
- `CLONE_TREE(treeId, baseUid)` —— **每实例一条**；双端按同一
  先序遍历从 `baseUid` 起连续分配元素 uid，无需传输任何 id 映射。

这两条是 Vue Lynx 自己的 opcode（主线程解释器与后台线程代码同 bundle 发布），
不是 Lynx 引擎变更。效果：创建 1,000 行列表时，Vapor 模式跨线程只需
**7,000 ops / 160 KB**，而 vdom 模式为 **17,000 ops / 327 KB**。

## 首屏直出（IFR）

Vapor 应用也可以在 Lynx `loadTemplate` 期间绘制同步内容。在 `vapor: true`
旁加上 `enableIFR: true`：

```ts title="lynx.config.ts"
pluginVueLynx({
  vapor: true,
  enableIFR: true,
})
```

Vapor IFR 复用已编译的 `template()` / 树协议——**不使用** VDOM 元素模板。
细节、取舍与 Vapor 特有限制见 [Vapor 上的 IFR](/zh/guide/ifr#ifr-on-vapor)。

## 当前限制

- **一个应用，一种模式。** Vue Lynx 不支持在同一应用里混用 vdom 与 Vapor
  （`vaporInteropPlugin` 尚未接入——上游将其硬编码到浏览器 DOM renderer）。
  模式是应用级严格二选一，因此 Vue Lynx 发布两个纯入口，而不是单入口靠
  tree-shaking 裁剪：

  | 入口 | 内容 | 用途 |
  |---|---|---|
  | `vue-lynx` | vdom renderer + Vue runtime-core | 常规（默认）应用 |
  | `vue-lynx/vapor` | Vapor 运行时 + Lynx DOM 兼容层 | Vapor 应用 |

  附带好处：vdom 专属 API 在 `vue-lynx/vapor` 上根本不存在——误用会**在
  构建期直接报错**。

- Vapor 应用中**暂无 `<Transition>` / `<Teleport>` / `<KeepAlive>` /
  `<Suspense>`** —— 上游 Vapor 尚未发布这些能力。
- **不支持 `v-html`** —— Lynx 元素无法解析 HTML 字符串（与 vdom 模式的
  Vue Lynx 一致）。
- **不支持 SSR / HTML hydration** —— Lynx 没有 SSR。（双线程
  [IFR hydration](/zh/guide/ifr#hydration) 是另一套机制。）
- Vapor 模式的常规 Vue 约束同样适用：仅 `<script setup>`、无 Options API、
  `getCurrentInstance()` 返回 `null`。
- Vapor IFR 要求首屏确定性、与线程无关。同步 `setup()` 副作用与 watcher 会
  在两端各跑一遍；见 [IFR 限制](/zh/guide/ifr#account-for-current-limitations)。
