# 基准测试：VDOM vs Vapor

[Vapor mode](/zh/guide/vapor-mode) 在 Lynx 上到底快多少？

> **[统一基准矩阵](/zh/guide/benchmark-unified)的一部分** ——
> 同机 IFR × VDOM/Vapor × React，含断言卡片与曲线。
> 本页只覆盖 *instrumented BG/e2e*（Vue VDOM vs Vapor）。

Vue Lynx 移植了 Vue 官方基准
（[`packages-private/benchmark`](https://github.com/vuejs/core/tree/main/packages-private/benchmark)，
派生自 [js-framework-benchmark](https://github.com/krausest/js-framework-benchmark)），
让**同一个应用以两种模式**跑完整的双线程管线。

**结论** *（instrumented lynx-web；看比率，不看绝对毫秒）*

| | 结果 | 备注 |
| --- | --- | --- |
| 更新路径（后台） | Vapor 比 VDOM 快 **5.8–9.8×** | 黑盒 selectStorm@10k 约复现 8× |
| 更新路径（端到端） | 快 **2.1–6.3×**（主线程 apply 成本共享） | 单次 select 常被帧地板掩盖 |
| 列表创建 | 跨线程 ops 约少 **2.4×**；端到端与 VDOM **持平** | 黑盒 1k→30k 约 1.0–1.1× |
| 代价 | 包体 gzip **+26%**；首屏差异在噪声内 | IFR 会改包体/FCP，见 IFR 页 |

想在手机上亲手感受同一批应用？见
[React vs Vue playground](/zh/guide/benchmark-playground)。

## 结果

### 更新路径

Vapor 的结构性优势——没有 vnode 树、没有 diff、响应式 effect 直写元素——在
后台线程上完整呈现：

| 操作 | vdom bg | vapor bg | bg 倍数 | vdom e2e | vapor e2e | e2e 倍数 |
|---|---|---|---|---|---|---|
| 每第 10 行更新 | 3.20 ms | 0.55 ms | **5.8×** | 4.85 ms | 2.30 ms | 2.1× |
| 选中行 | 2.95 ms | 0.30 ms | **9.8×** | 3.45 ms | 0.55 ms | 6.3× |
| 交换行 | 3.90 ms | 0.65 ms | **6.0×** | 4.65 ms | 1.20 ms | 3.9× |
| 删除行 | 2.85 ms | 0.40 ms | **7.1×** | 3.40 ms | 0.90 ms | 3.8× |

e2e 倍数小于 bg，因为这些操作两种模式发出的 ops 几乎相同——差距全部来自
后台线程上的 Vue 工作量，而那正是与你的业务逻辑共享的线程。

### 创建路径

| 操作 | vdom e2e | vapor e2e | 比值 |
|---|---|---|---|
| 创建 1,000 行 | 125.7 ms | 144.5 ms | 0.87×（vapor 慢 15%） |
| 追加 1,000 行 | 217.4 ms | 224.6 ms | 0.97× |
| 创建 10,000 行 | 1500 ms | 1602 ms | 0.94× |
| 清空 10,000 行 | 1838 ms | 1621 ms | **1.13×（vapor 更快）** |

创建路径已基本持平。达到这一步依赖 `REGISTER_TREE` / `CLONE_TREE` 协议与
纯 `vue-lynx/vapor` 入口（见 [Vapor mode](/zh/guide/vapor-mode#how-it-works-on-lynx)）。

### 跨线程流量

每次创建 1k 行：

| | vdom | vapor |
|---|---|---|
| 每次 flush 的 ops | 17,000 | **7,000（−59%）** |
| JSON 字节数 | 327 KB | **160 KB（−51%）** |

在 Web harness 上这几乎不再压低 e2e（主线程被真实 DOM 构建主导）。在原生
Lynx 上载荷要跨序列化边界进入 PrimJS——减半在真机上预期更有意义。

### 首屏、内存、包体

| 指标 | vdom | vapor | 备注 |
|---|---|---|---|
| 首屏（lynx-view attach → 首次内容） | 121.7 ms | 125.8 ms | +3%，置信区间重叠 |
| 包体 gzip（`main.lynx.bundle`） | 39.2 KB | 49.3 KB | +26% —— Vapor 运行时 + DOM 兼容 shim |
| 10k 行后 JS 堆 | 76–136 MB | 76–105 MB | 仅作指示（未强制 GC） |

## 方法

- **应用**：krausest 表格应用（对 1k–10k 行执行
  create/update/select/swap/remove/append/clear）。Vapor 版由 vdom 源码
  *生成*——唯一差异是 `vapor` 属性——负载逐字节一致。
- **Harness**：生产构建，经
  [Lynx for Web](https://lynxjs.org/guide/start/fragments/web/)（`<lynx-view>`）
  在 headless Chromium（Playwright）中渲染。Vue 在后台 worker，ops 解释器 +
  DOM 在主线程。
- **指标**：
  - `bg` —— 后台线程成本：响应式 + 渲染 + ops 序列化
  - `e2e` —— `bg` + 跨线程传输 + 主线程 apply
  - ops 流遥测（条数 + JSON 字节）、首屏、JS 堆、包体
- **统计**：每操作 n=20（重操作 n=10），中位数 ± 95% CI，两种模式交替加载，
  另跑独立复现轮。

## 注意事项

- Harness 是 **Lynx for Web，不是真机**。`bg` 在结构上与引擎无关；e2e 与首屏
  在原生 Lynx（PrimJS、原生排版）上会不同。同一 bundle 可进 LynxExplorer 复测。
- 亚毫秒级 e2e 中位数接近计时粒度；bg 中位数 CI95 &lt; 15%，小操作的 e2e CI
  较宽。
- Vapor mode 与 Vue 3.6 均为 prerelease；数字会变化。
