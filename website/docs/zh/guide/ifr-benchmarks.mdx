## 1. 策略阶梯

> **结论：元素模板是渲染成本的拐点。**
> 纯 IFR 重放 ≈ 完整渲染 JS（解释器下约 8–11 ms / 千元素）；ET 砍 7–15×；
> Vapor 式设计还能再靠近 PAPI 地板 2–3×。所以 ET 默认随 IFR 开启。

七种策略、同样逻辑首屏、文档逐字节一致。场景约 1000–1400 元素：
**static-heavy**（99.7% 静态）、**content**（37%）、**list**（12%）。

### Warm 渲染（`--jitless` V8 ≈ 主线程引擎代理）

| variant | static-heavy | content | list |
|---|---|---|---|
| bg-baseline（No IFR） | 11.86 ms | 9.36 ms | 6.87 ms |
| IFR without ET | 10.94 ms | 8.40 ms | 6.57 ms |
| ifr-direct（原型） | 8.17 ms | 6.36 ms | 4.89 ms |
| ifr-static-tpl（原型） | 1.11 ms | 5.29 ms | 4.00 ms |
| **IFR + ET（默认）** | **0.74 ms** | **1.26 ms** | **1.44 ms** |
| ifr-vapor（上界） | 0.53 ms | 0.55 ms | 0.46 ms |
| papi-floor | 0.54 ms | 0.39 ms | 0.31 ms |

### Cold 首次（`--jitless`）

| variant | static-heavy | content | list |
|---|---|---|---|
| IFR without ET | 18.0 ms | 16.4 ms | 11.7 ms |
| **IFR + ET** | **4.2 ms** | **5.3 ms** | **4.9 ms** |

### 跨线程 ops 载荷

## 2. 全示例扫描

> **结论：测成本与语义安全——不是产品 FCP 收益。**
> Bundle gzip **×2.26**，TTI 上界 **×1.36**；22/23 示例跨配置逐字节一致。
> 单进程 harness 里逐项 FCP **持平**（中位 ×1.04）——IFR 要有真实线程边界
> 才显收益（见统一矩阵 / VERIFICATION.md）。

## 3. Native 引擎观察

LynxExplorer（SDK 1.4 / PrimJS）：24/25 示例 **零 hydration mismatch**；
No-IFR vs IFR+ET 视觉一致（SSIM ≥ 0.9977）。单次冷启动：IFR 首内容约早
0.3 s（gallery）到 1.2 s（7guis）——方向性确认，不是稳定基准。
