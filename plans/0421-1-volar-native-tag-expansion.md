# Volar Plugin: Native Tag Expansion & IDE Diagnostics

## Scope

Three DX improvements to the `vue-lynx` Volar plugin and element types:

1. All Lynx built-in elements registered as native tags — eliminates "Unknown component" IDE errors
2. Unsupported event modifier diagnostic for `.capture` / `.passive`
3. `global-bind*`, `global-catch*`, `main-thread-*`, and `main-thread-ref` recognised as valid props

## Problem

The Volar plugin previously only declared three native tags: `block`, `template`, `slot`. Every real Lynx element — `<view>`, `<text>`, `<image>`, `<scroll-view>`, etc. — was flagged as "Unknown component" in VS Code with Volar installed, even though `@lynx-js/types` already provided full IntrinsicElements type coverage.

The mismatch was: types declared in `GlobalComponents` (recognised by TypeScript) but `isNativeTag` / `nativeTags` unknown to Volar's template compiler, which has a separate native-tag concept.

## Why `require('@lynx-js/types')` Does Not Work

`@lynx-js/types` is a types-only package — its `exports` map exposes only `"types"` conditions, with no runtime JS. Calling `require('@lynx-js/types')` at Volar tooling time returns nothing useful.

## Solution

**Static list + codegen script.**

The `nativeTags` array in the plugin is split into:

```js
const metaTags = ['block', 'template', 'slot']; // Vue template compilation
const lynxTags = [                               // from @lynx-js/types IntrinsicElements
    'component', 'filter-image', 'image', 'inline-image',
    'inline-text', 'inline-truncation', 'list', 'list-item',
    'list-row', 'page', 'scroll-view', 'text', 'view',
    'raw-text', 'input', 'textarea', 'frame', 'overlay', 'svg',
];
const nativeTags = [...metaTags, ...lynxTags];
```

A `scripts/generate-native-tags.mjs` script reads `@lynx-js/types/types/common/element/element.d.ts`, extracts `IntrinsicElements` keys via regex, and rewrites the `lynxTags` array in `src/volar-plugin.cjs`. Run it after upgrading `@lynx-js/types`:

```
pnpm generate:native-tags   # from packages/vue-lynx/types
```

## Modifier Diagnostic

`lynxModifierTransform` is a `nodeTransforms` entry injected via `resolveTemplateCompilerOptions` (Volar >= 2.0.14). It walks `ELEMENT` nodes, finds `v-on` directive nodes, and calls `context.onError` for any `.capture` or `.passive` modifier:

```js
const ELEMENT_NODE = 1;   // NodeTypes.ELEMENT
const DIRECTIVE_NODE = 7; // NodeTypes.DIRECTIVE

function lynxModifierTransform(node, context) {
  if (node.type !== ELEMENT_NODE) return;
  for (const prop of node.props) {
    if (prop.type !== DIRECTIVE_NODE || prop.name !== 'on') continue;
    for (const modifier of prop.modifiers) {
      // string[] in Vue <3.3; SimpleExpressionNode[] in Vue >=3.3
      const name = typeof modifier === 'string' ? modifier : modifier.content;
      if (!['capture', 'passive'].includes(name)) continue;
      const loc = (typeof modifier === 'object' && modifier.loc) ? modifier.loc : prop.loc;
      const err = new SyntaxError(`Lynx does not support the .${name} event modifier.`);
      err.code = 1001;
      err.loc = loc;
      context.onError(err);
    }
  }
}
```

NodeType constants are hardcoded to avoid `require('@vue/compiler-core')` at IDE time.

## Special Prop Types

`LynxSpecialProps` added to `VueLynxProps<T>` in `elements/index.ts`:

```ts
type LynxSpecialProps = {
  'main-thread-ref'?: MainThreadRefLike | null;
  [key: `global-bind${string}`]: Function | undefined;
  [key: `global-catch${string}`]: Function | undefined;
  [key: `main-thread-${string}`]: unknown;
}
```

Template literal and mapped props suppress "unknown prop" errors for Vue Lynx's runtime-supported hyphenated syntax and surface in TypeScript completions.

## Testing in the IDE

### 1. Native tag no-error
Open any `.vue` file, use `<view>`, `<text>`, `<image>`, `<scroll-view>`, etc. — no "Unknown component" squiggle. Hover shows prop completions from `@lynx-js/types`.

### 2. Modifier diagnostic
```vue
<view @tap.capture="handler" />
```
Volar should show a red squiggle on `.capture` with the message: *"Lynx does not support the .capture event modifier."*

### 3. global-bind / global-catch
```vue
<view :global-bindtap="handler" />
<view :global-catchtap="handler" />
```
No "unknown prop" error. The value type resolves to the corresponding event handler type.

### 4. main-thread-ref
```vue
<view :main-thread-ref="myEl" />
```
No "unknown prop" error. `main-thread-ref` accepts the object returned by `useMainThreadRef()`.

### 5. main-thread worklet events
```vue
<view :main-thread-bindtap="handler" />
```
No "unknown prop" error (covered by the `main-thread-*` prop mapping).
