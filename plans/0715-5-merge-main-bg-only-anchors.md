# Merging main into `vapor`: BG-only anchors adopted

Date: 2026-07-15. The `vapor` branch (PR #222) had drifted from `main` by
five substantive commits (worklet allowlist #190, global events #193,
v-model native timing #203, empty-anchor fix #201, go-web/web-core bumps
#207/#221, plus the ai-chat/elk examples). Most of the merge was
mechanical; two pieces were architectural.

## The anchor-design fork — decided on correctness, not cost

Both branches had independently fixed the same bug (#201's "empty
raw-text gets a default line box on native"), with opposite designs:

- **vapor branch**: materialize every anchor on the Main Thread as a
  hidden element — comments as `display:none` views, text nodes as
  `display:none` raw-text toggled per SET_TEXT. Uniform 1:1 tree mirror;
  verified 45/45 visual parity on web.
- **main**: comments and empty `#text` nodes never reach the Main Thread
  at all. Text materializes lazily while it has content; inserts resolve
  their anchor by walking to the next materialized sibling.

Main's design won (the user's directive: "不要去思考 cost of work，要思考
哪个架构更正确"):

1. The MT tree should contain exactly the renderable elements — anchors
   are renderer bookkeeping and belong to the BG shadow tree.
2. `display:none` on native raw-text is exactly the kind of styling #201
   showed to be unreliable; web-only verification couldn't prove it safe.
3. One native element per v-if/Fragment anchor plus one style op per text
   toggle is pure overhead, and it skewed the ops-stream numbers the
   benchmarks report.

### What adoption looked like

- `tree-ops.ts` gained the materialization model shared by both
  renderers: `isMaterialized`, `resolveMainThreadAnchor`, `setTextNode`
  (lazy create/SET_TEXT/insert/remove), and materialization-aware
  `insertNode`/`removeNode`/`setElementTextContent`. `_mtCreated` /
  `_mtInserted` live on ShadowElement; `_text` is the single text field
  (main's duplicate `_textValue` dropped).
- vdom `node-ops.ts`: `createComment` is shadow-only; `createText`
  pushes ops only for non-empty text; `setText` delegates to
  `setTextNode`.
- Vapor: granular clones skip anchor ops; `_materializeAliasedText`
  stays lazy for empty aliased text; **the template uid contract is
  preserved by consuming uids without creating elements** — MT
  instantiation returns null for `#comment` / empty `#text` structure
  nodes but still advances the pre-order counter, and the BG shadow
  clone marks those nodes unmaterialized. No protocol change.
- MT interpreter: `__comment` CREATE handling, CREATE_TEXT hiding, and
  SET_TEXT display toggling all deleted.
- Main's three tree-shape tests (testing-library) kept **as-is** and now
  pass; the MT template spec re-asserts anchors-off-MT and the uid-gap
  contract.

Note for old bundles: the `'__comment'` CREATE argument was never an
opcode, just a tag string — no opcode is retired by this.

## Dependency patches re-ported, not dropped

Main bumped `@lynx-js/go-web` 0.2.1→0.6.0 and `@lynx-js/web-core`
0.20.2→0.22.1 and deleted the pnpm patches. Upstream absorbed only part
of what the patches carried:

- go-web 0.6.0 ships `key={src}` (remount on source change) natively,
  but NOT `exampleMetadata` preload / `onEntryChange` — which the
  website's render-mode store depends on. Patch re-created for 0.6.0
  with just those.
- web-core 0.22.1 still has **both** unguarded crash paths: the
  `loadScript` detached-realm crash (provoked by the renderer switch's
  `key={mode}` remount) and the `callDestroyLifetimeFun` dispose
  TypeError. Patch re-created for 0.22.1 unchanged.

> Rule from `0715-4` still holds: when an upstream bump drops a patch,
> diff the new version for each guard the patch carried before believing
> the bump fixed anything.
