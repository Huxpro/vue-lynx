The <list> element: foundational PAPI contracts & which framework they fit

A ground-up map of the Main-Thread Element PAPI contracts that make a native Lynx <list> render, scroll, diff, and recycle — from the two-call basics to the full recycling protocol — followed by an analysis of what those contracts reward and punish in React, Vue, and signal-based frameworks.

Part I · 0Mental model: the native list virtualizes; the framework is a data source

Unlike a <scroll-view>, a Lynx <list> does not hold one element per data row. The native layer owns virtualization: it decides which cells are on-screen and, when it needs one, it calls back into JS to materialize that cell's element subtree. The framework never "renders the list"; it registers a set of callbacks and keeps a side table of item metadata, and the native list pulls from them on demand.

Everything below is an elaboration of that inversion of control. Three things flow across the boundary:

  • Callbacks (JS → registered on the element): componentAtIndex, enqueueComponent, componentAtIndexes — native invokes these to mount, recycle, and batch-mount cells.
  • The update-list-info attribute (JS → native): a declarative diff of the data source (insert / remove / update actions keyed by item-key).
  • Platform-info attributes (per item): item-key, reuse-identifier, sticky-top, full-span, estimated-main-axis-size-px — read by native, never applied as ordinary element attributes.
The one idea to hold onto

The list is a protocol between a virtualizing native widget and a JS data source, not a container you fill with children. A framework "supports <list>" to exactly the degree that its rendering model can be turned inside-out into pull-based, per-cell, recyclable production.

Part I · 1  tier 1 · basicsBasics — create the element & feed items

A native list is created with __CreateList, which is unique among element-creation PAPIs in that it takes three callbacks at construction time. The minimum viable contract is componentAtIndex: given a cell index, append the element subtree for that row to the list and return its unique id (the uiSign).

The PAPIs at this tier

REPL-executable: a static list

main-thread.jsL0 · vanilla PAPI
// A minimal <list>: native pulls each visible row via componentAtIndex.
const page = __CreatePage("0", 0);
const rows = ["Alpha", "Bravo", "Charlie", "Delta"];

function componentAtIndex(list, listID, cellIndex, operationID) {
  const item = __CreateElement("list-item", 0);
  __SetAttribute(item, "item-key", "row-" + cellIndex);   // platform-info, required
  const text = __CreateElement("text", 0);
  __SetAttribute(text, "text", rows[cellIndex]);
  __AppendElement(item, text);
  __AppendElement(list, item);                          // attach under the list
  const sign = __GetElementUniqueID(item);
  __FlushElementTree(item, { triggerLayout: true, operationID, elementID: sign, listID });
  return sign;                                          // native tracks the cell by this uiSign
}

const list = __CreateList(0, componentAtIndex, function(){}, function(){});
__SetAttribute(list, "list-type", "single");
__SetAttribute(list, "span-count", 1);
__AppendElement(page, list);
__FlushElementTree(page);

At this tier enqueueComponent and componentAtIndexes are stubbed to no-ops. But there is a hidden problem: the list learns how many rows exist, and any change to that count or ordering, only through the update-list-info attribute. That is tier 2.

Part I · 2  tier 2 · mutationMutation — the update-list-info diff contract

The native list keeps its own model of the data source. You mutate that model declaratively by setting the update-list-info attribute with three arrays. This is the contract the PR-stack's merge-order fix is built on.

the payload shape
__SetAttribute(list, "update-list-info", {
  insertAction: [{ position: 2, type: "list-item", "item-key": "c", /* ...platform info */ }],
  removeAction: [1, 4],           // OLD indices into the pre-diff snapshot, sorted ascending
  updateAction: [{ from: 0, to: 0, type: "list-item", flush: false, /* platform info */ }],
});
  • insertAction — new cells, each with a position (target index) and the item's platform info folded in.
  • removeAction — indices into the previous snapshot. Both ReactLynx's ListUpdateInfoRecording.__toAttribute() and the vue-lynx adapter emit these sorted numerically; native applies them against the pre-diff snapshot as a batch.
  • updateAction — cells whose platform info changed but which stayed in place (from/to).
Why a diff, not per-op mutation

Re-declaring the same item-key makes the native list error 2202 (duplicated item-key). So the framework cannot naively "append what's new" — it must compute a genuine diff between the last flushed order and the current order, expressing moves as remove+insert. Both runtimes model this the same way (ReactLynx: "move = removeChild + insertBefore").

How the two runtimes compute the diff

ReactLynx records mutations imperatively as its reconciler walks the tree (insertBefore map + appendChild list + removals set), because the reconciler already visits every change. The vue-lynx adapter has no such hook, so it keeps a lastFlushed snapshot and computes the diff structurally with a Longest Increasing Subsequence over old indices — the LIS is the set of items that stayed; everything else is remove+insert:

vue-lynx · list-apply.ts (diffListItems)
// common items, expressed as their OLD indices in NEW order
const commonNewOrder = newItems
  .map(e => oldIndex.get(e.bgId))
  .filter(i => i !== undefined);
const stayed = longestIncreasingSubsequence(commonNewOrder); // keep these
// removeAction = old indices not in `stayed`;  insertAction = new items not in `stayed`

The results are equivalent, but the source of the diff differs sharply — and that difference is the whole framework-fit story in Part II. React gets the diff for free from reconciliation; Vue reconstructs it after the fact.

Part I · 3  tier 3 · batchBatch & async rendering

Pulling cells one componentAtIndex call at a time is fine for a handful of visible rows, but native wants to warm several cells (the preload-buffer-count) in one hop across the thread boundary, and to defer layout. Two mechanisms:

MechanismSignature / flagRole
componentAtIndexes(list, listID, cellIndexes[], operationIDs[], enableReuseNotification?)Batch form: mount many cells, then one __FlushElementTree(list, …) for the whole batch.
operationIDnumberCorrelates a specific native pull with the flush that satisfies it, so native can match async results.
asyncFlush__FlushElementTree(el, {asyncFlush:true})Lets layout happen off the critical path when batch-rendering is enabled.
enableBatchRenderruntime capabilityWhen set, per-cell layout flushes are suppressed in favor of the batch flush.
main-thread.js · batch mountL0
function componentAtIndexes(list, listID, cellIndexes, operationIDs) {
  const ids = [];
  for (const idx of cellIndexes) ids.push(mountCell(list, listID, idx)); // no per-cell flush
  __FlushElementTree(list, { triggerLayout: true, operationIDs, elementIDs: ids, listID });
  return ids;
}

These are throughput optimizations, not new semantics — but note that operationID plumbing is threaded through every mount path, which becomes a source of complexity in the recycling tier.

Part I · 4  tier 4 · recyclingRecycling — the pool protocol

This is where the contract stops being a simple pull and becomes stateful. When a cell scrolls off-screen native calls enqueueComponent(list, listID, uiSign). Instead of destroying that subtree, the framework pools it under its reuse-identifier. The next componentAtIndex for a compatible cell should reuse a pooled subtree rather than build a fresh one — mutating only the parts that differ.

The two-map state (both runtimes)

StructureReactLynxvue-lynxKeyed by
Active cellsgSignMapsignMapslistID → (uiSign → entry)
Recycle poolgRecycleMaprecycleMapslistID → (reuseKey → (uiSign → entry))

The componentAtIndex mount path becomes a three-way decision:

  1. Self-reuse — the same item scrolled back into view; its own uiSign is still in the pool. Pop it, re-append, done. Cheapest path, no content mutation.
  2. Cross-item reuse — a different item of the same reuse-identifier is pooled. Take that subtree and hydrate it with the new cell's data.
  3. Fresh mount — pool empty; build a new subtree (tier 1 path).
The decisive divergence: what "hydrate" means

ReactLynx calls the snapshot's hydrate(oldSnapshot, newSnapshot) — a compiler-generated function that knows exactly which dynamic slots in the template differ and patches only those. Recycling is a first-class product of its snapshot model.

vue-lynx has no per-template hydrate primitive, so hydrateListItemEntry() does it structurally at runtime: build a reverse element→id index, walk donor and target subtrees in lockstep, swap textContent/attributes and exchange element-id ownership node-by-node. Correct for isomorphic cells, O(tree) per recycle, and fragile if two cells of the same reuse-identifier differ in shape (a v-if branch). See PR #307 review.

REPL-executable: a minimal recycling pool

main-thread.js · self-reuse + poolL0
// Minimal tier-4 sketch: pool by reuse-identifier, prefer self-reuse.
const signMap = new Map();               // uiSign -> { el, key }  (active)
const pool    = new Map();               // reuseId -> Map(uiSign -> { el, key })

function componentAtIndex(list, listID, cellIndex, operationID) {
  const reuseId = reuseIdOf(cellIndex);
  const bucket  = pool.get(reuseId);
  let entry;
  if (bucket && bucket.size) {                     // reuse a pooled subtree...
    const [sign, pooled] = bucket.entries().next().value;
    bucket.delete(sign);
    hydrate(pooled.el, cellIndex);                // ...only patch what differs
    entry = pooled;
  } else {
    entry = { el: buildCell(cellIndex), key: keyOf(cellIndex) }; // fresh mount
  }
  __AppendElement(list, entry.el);
  const sign = __GetElementUniqueID(entry.el);
  signMap.set(sign, entry);
  __FlushElementTree(entry.el, { triggerLayout: true, operationID, elementID: sign, listID });
  return sign;
}

function enqueueComponent(list, listID, sign) {   // cell scrolled off-screen
  const entry = signMap.get(sign);
  if (!entry) return;
  signMap.delete(sign);
  const reuseId = reuseIdOfKey(entry.key);
  if (!pool.has(reuseId)) pool.set(reuseId, new Map());
  pool.get(reuseId).set(sign, entry);             // park it, don't destroy
}
// hydrate() is the crux — see the note below on what it means per framework.

The recycling PAPIs & lifecycle

PAPISignatureRole
enqueueComponent(list, listID, uiSign) → voidNative hands a scrolled-off cell back; move it from the active map into the reuse pool.
reuse-identifierattr on list-itemPool key — only cells sharing it may cross-item-reuse each other.
__UpdateListCallbacks(list, componentAtIndex, enqueueComponent, componentAtIndexes)Re-bind the three callbacks so native holds fresh closures over current data. ReactLynx does this on every flush; vue-lynx followed suit in #311.
destroyinstall inert callbacks + clear mapsOn list teardown, ReactLynx's snapshotDestroyList installs no-op callbacks (() => -1) and clears bookkeeping; vue-lynx's destroyListElement mirrors it (#311).
Why __UpdateListCallbacks must fire every flush

The callbacks close over the item table. In a snapshot model the closure must be refreshed so native pulls see the new data — hence ReactLynx's every-flush rebind. In vue-lynx the closures read live Maps, so the rebind is closer to parity-for-parity's-sake than a functional necessity (PR #311 review) — a small but telling sign that the contract was shaped around the snapshot model.

Part II · 5What the contract actually demands of a framework

Strip the PAPIs down to the capabilities they require of whatever framework sits above them. A framework adapting <list> well must provide, in rough order of difficulty:

  1. Pull-based, index-addressable rendering. Produce the subtree for row N on demand, out of document order. Most declarative frameworks render a whole subtree top-down, not one indexed child in isolation.
  2. A stable, computable order diff. Turn "the data went from order A to order B" into insert/remove/move against keys — ideally for free from whatever the framework already does to reconcile.
  3. A cheap re-target / hydrate primitive. Take an existing subtree built for item X and make it display item Y by mutating only the differences — the core of cross-item recycling.
  4. Stable identity across the thread boundary. A durable mapping between framework-level nodes and native uiSigns that survives moves, pooling, and re-target.

Capabilities 2 and 3 are the discriminating ones: they are exactly what a snapshot/VDOM-with-keyed-reconciliation model produces as a by-product, and exactly what other models must bolt on.

Part II · 6React / Vue / signals, compared against the contract

ReactLynx native fit

Snapshot model + keyed reconciliation. The contract was co-designed with it.

Gains
  • The reconciler is the diff — insertBefore/removal recording is a free by-product.
  • Compiler-generated hydrate() per snapshot → precise, cheap cross-item re-target.
  • Snapshot instances are natural pool entries with stable identity.
Costs
  • Requires the whole snapshot compiler + runtime; not a thin adapter.
  • Callbacks close over data → must rebind every flush.

Vue (vue-lynx) workable, with seams

Custom renderer over a reactive VNode tree; no compile-time snapshot.

Gains
  • Reactivity makes the data source side trivial and fine-grained.
  • Custom renderer cleanly intercepts insert/remove into the item table.
  • Live Map closures sidestep some rebind pressure.
Costs
  • No free order-diff → must reconstruct it with an LIS over snapshots (#293).
  • No hydrate primitive → structural runtime attr-walk, O(tree) & shape-fragile (#307).
  • Identity remap across pooling is hand-maintained (reverse index rebuild).

Signals (Solid-like) promising

Fine-grained reactivity, compiled templates, explicit <For>.

Gains
  • Per-cell reactive scopes map almost 1:1 onto per-cell pull + re-target.
  • A cell is a reactive computation; "hydrate" = swap the row signal, DOM updates itself. This is the cheapest possible cross-item reuse.
  • <For> already maintains keyed identity for reconciliation.
Costs
  • Must express order changes as the same insert/remove diff.
  • Pull-based, out-of-order mount fights the usual top-down create.

Svelte / compiled mixed

Compiled surgical updates, keyed {#each} blocks.

Gains
  • Compiled update fns give a hydrate-like precise re-target, like Solid.
  • Keyed each-blocks carry identity.
Costs
  • Block-oriented, not index-addressable — pull-per-cell needs adaptation.
  • Component boundaries are compiled away, complicating pool entries.
The pattern across all four

The single capability that separates "native fit" from "workable with seams" is a per-template re-target/hydrate primitive. Frameworks that compile templates (React-snapshot, Solid, Svelte) get it for free and recycle precisely. Frameworks that interpret a runtime tree (Vue's VDOM) must emulate it structurally — which is exactly where vue-lynx's #307 risk lives.

Part II · 7 · direction AWhich framework best fits the current <list> API?

As it stands, the API rewards a compile-time-template, keyed-reconciliation model — and ReactLynx fits it best because the two were designed together. Every hard requirement of the contract (free order-diff, compiler hydrate, snapshot pool entries, every-flush rebind) lines up with a feature the ReactLynx snapshot runtime already has. That is not a neutral coincidence: update-list-info's remove+insert move semantics, the reuse-identifier pool keying, and the rebind-on-flush lifecycle all read like they were extracted from the snapshot reconciler.

The most elegant theoretical fit, though, is fine-grained signals (Solid-style): when a cell is a reactive computation, cross-item recycling collapses to "reassign the row's source signal and let the effects patch the DOM" — no structural walk, no manual identity remap, cheaper than even ReactLynx's compiled hydrate. Signals fit the spirit of pull-based recycling better than React fits the letter of the current API.

Vue sits in the middle by construction. Its reactivity makes the data-source half delightful, but its runtime VDOM gives it neither the free diff nor the hydrate primitive, so the adapter must reconstruct both — the LIS diff and the structural attr-walk. It is genuinely workable (the PR stack proves the happy path), but the seams (#307's O(tree) fragile hydrate, #311's semantically-redundant rebind) are the direct price of adapting a runtime-tree framework to a snapshot-shaped contract.

Part II · 8 · direction BHow could the <list> API be redesigned to better support frameworks?

The current contract leaks one implementation's internals (the snapshot reconciler) into the boundary. A more framework-neutral contract would let each framework contribute the piece it is good at and stop forcing the pieces it is not. Concretely:

Friction in today's APIWho it hurtsRedesign direction
Framework must supply an order-diff (update-list-info) even though native also tracks order and even emits a diffResult in layoutcomplete.Non-reconciling frameworks (Vue, imperative).Accept a keyed snapshot of the current order and let native compute the move-diff itself; make diffResult a first-class input path, not just telemetry.
Cross-item recycling assumes a hydrate(old,new) exists; frameworks without one must fake it.Runtime-tree frameworks (Vue).Add an explicit recycleComponent(list, uiSign, newCellIndex) callback so the framework re-targets a subtree using its own update path — no structural walking imposed.
Callbacks close over data → must rebind every flush.Everyone (needless churn).Pass current data as an argument to the pull (componentAtIndex(list, listID, idx, opID, ctx)) so callbacks stay identity-stable and rebind becomes rare.
Identity is a raw native uiSign the framework must map back to its own node.Everyone.Let the framework register an opaque handle alongside the uiSign so native round-trips it back on enqueue/recycle — no reverse index to rebuild.
Platform info is stringly-typed and folded into insertAction.Type-safe frameworks.A typed ListItemInfo struct separate from the diff payload.

The unifying principle: the boundary should ask frameworks for data + identity, and ask them to perform re-target through their own machinery — not to reproduce a specific reconciler's diff and hydrate. Today the contract says "be a snapshot reconciler." A framework-neutral contract would say "give me your keys and a way to update a cell, and I'll drive." That change would make Vue's adapter shed both its LIS diff and its structural hydrate, would let a signals framework plug its per-cell effects straight in, and would cost ReactLynx almost nothing (it can still compute the diff and hydrate internally if it prefers).

Bottom line

Direction A: the current API fits ReactLynx best (by design) and suits Solid-style signals most elegantly; Vue is workable but pays for the impedance mismatch at exactly the two points the PR stack had to engineer around. Direction B: move order-diffing and identity-tracking to native, and replace the implicit hydrate assumption with an explicit recycleComponent re-target callback — turning a snapshot-shaped contract into a data-and-identity contract that any keyed framework can satisfy with its own update path.

第一部分 · 0心智模型：原生 list 负责虚拟化，框架只是数据源

与 <scroll-view> 不同，Lynx 的 <list> 并不为每一行数据持有一个元素。虚拟化由原生层掌控：它决定哪些 cell 在屏幕上，当它需要某个 cell 时，会回调进 JS 来实例化该 cell 的元素子树。框架从不「渲染整个 list」，它只是注册一组回调并维护一张item 元数据侧表，原生 list 按需从中拉取。

下文全部都是对这一「控制反转」的展开。跨越边界流动的有三样东西：

  • 回调（JS → 注册到元素上）：componentAtIndex、enqueueComponent、componentAtIndexes——原生借此挂载、回收、批量挂载 cell。
  • update-list-info 属性（JS → 原生）：对数据源的声明式 diff（以 item-key 为键的 insert / remove / update）。
  • platform-info 属性（逐 item）：item-key、reuse-identifier、sticky-top、full-span、estimated-main-axis-size-px——由原生读取，绝不作为普通元素属性下发。
唯一要抓住的核心

list 是一个「虚拟化原生控件」与「JS 数据源」之间的协议，而不是一个你往里塞子节点的容器。一个框架能在多大程度上「支持 <list>」，完全取决于它的渲染模型能否被翻转成拉取式、逐 cell、可回收的生产方式。

第一部分 · 1  层级 1 · 基础基础——创建元素并喂入 item

原生 list 由 __CreateList 创建，它在所有「元素创建」PAPI 中很特殊——构造时就要传入三个回调。最小可用契约是 componentAtIndex：给定 cell 索引，把该行的元素子树挂到 list 上，并返回它的 unique id（即 uiSign）。

第一部分 · 2  层级 2 · 变更变更——update-list-info diff 契约

原生 list 自己维护一份数据源模型。你通过设置 update-list-info 属性、传三个数组来声明式地变更这份模型。这正是本 PR 栈「合并顺序修复」所依赖的契约。

payload 结构
__SetAttribute(list, "update-list-info", {
  insertAction: [{ position: 2, type: "list-item", "item-key": "c", /* ...platform info */ }],
  removeAction: [1, 4],           // 指向 diff 前快照的「旧索引」，升序排列
  updateAction: [{ from: 0, to: 0, type: "list-item", flush: false, /* platform info */ }],
});
  • insertAction——新 cell，各带一个 position（目标索引）并把该 item 的 platform info 合并进来。
  • removeAction——指向上一份快照的索引。ReactLynx 的 ListUpdateInfoRecording.__toAttribute() 与 vue-lynx 适配层都按数值升序输出，原生把它们作为一批、相对 diff 前快照来应用。
  • updateAction——platform info 变了但位置未变的 cell（from/to）。
为什么是 diff，而不是逐操作变更

重复声明同一个 item-key 会让原生 list 报 2202（duplicated item-key）。因此框架不能天真地「把新增的追加上去」——它必须在「上次 flush 的顺序」和「当前顺序」之间算出真正的 diff，把移动表达为 remove+insert。两套运行时的建模一致（ReactLynx：「move = removeChild + insertBefore」）。

两套运行时如何算 diff

ReactLynx 在 reconciler 遍历树时命令式地记录变更（insertBefore map + appendChild 列表 + removals 集合），因为 reconciler 本就访问每一处改动。vue-lynx 适配层没有这种钩子，于是它保留一份 lastFlushed 快照，用「旧索引」上的最长递增子序列（LIS）结构性地算 diff——LIS 即「留在原处」的集合，其余皆为 remove+insert：

vue-lynx · list-apply.ts (diffListItems)
// 共有 item，用其「新顺序中的旧索引」表达
const commonNewOrder = newItems
  .map(e => oldIndex.get(e.bgId))
  .filter(i => i !== undefined);
const stayed = longestIncreasingSubsequence(commonNewOrder); // 保留这些
// removeAction = 不在 stayed 里的旧索引；insertAction = 不在 stayed 里的新 item

结果等价，但 diff 的来源迥异——而这个差异正是第二部分的全部戏剧。React 从 reconciliation 里免费拿到 diff；Vue 事后重建它。

第一部分 · 3  层级 3 · 批量批量与异步渲染

一次 componentAtIndex 拉一个 cell，对少量可见行没问题；但原生希望一次跨线程就预热多个 cell（即 preload-buffer-count）、并延后布局。两个机制：

机制签名 / 标志职责
componentAtIndexes(list, listID, cellIndexes[], operationIDs[], enableReuseNotification?)批量形式：挂载多个 cell，再对整批做一次 __FlushElementTree(list, …)。
operationIDnumber把某次原生拉取与满足它的 flush 关联起来，便于原生匹配异步结果。
asyncFlush__FlushElementTree(el, {asyncFlush:true})启用批量渲染时，让布局离开关键路径。
enableBatchRender运行时能力开启后，逐 cell 的布局 flush 被抑制，改用批量 flush。
main-thread.js · 批量挂载L0
function componentAtIndexes(list, listID, cellIndexes, operationIDs) {
  const ids = [];
  for (const idx of cellIndexes) ids.push(mountCell(list, listID, idx)); // 不逐 cell flush
  __FlushElementTree(list, { triggerLayout: true, operationIDs, elementIDs: ids, listID });
  return ids;
}

这些是吞吐优化，而非新语义——但要注意 operationID 会贯穿每一条挂载路径，这会在回收层级成为复杂度来源。

第一部分 · 4  层级 4 · 回收回收——池化协议

到这里契约不再是简单拉取，而变得有状态。当 cell 滚出屏幕，原生调用 enqueueComponent(list, listID, uiSign)。框架不销毁该子树，而是把它按 reuse-identifier 池化。下一次针对兼容 cell 的 componentAtIndex 应当复用池中子树而非重建——只改动有差异的部分。

两张 map 状态（两套运行时一致）

结构ReactLynxvue-lynx键
活跃 cellgSignMapsignMapslistID → (uiSign → entry)
回收池gRecycleMaprecycleMapslistID → (reuseKey → (uiSign → entry))

componentAtIndex 的挂载路径变成三岔决策：

  1. 自复用——同一 item 又滚回可见区，它自己的 uiSign 还在池里。弹出、重新挂载即可。最便宜，无需改内容。
  2. 跨 item 复用——池里有另一个同 reuse-identifier 的 item。取那棵子树，用新 cell 的数据hydrate它。
  3. 全新挂载——池空，构建新子树（层级 1 路径）。
决定性分歧：「hydrate」到底是什么

ReactLynx 调用 snapshot 的 hydrate(oldSnapshot, newSnapshot)——一个编译器生成的函数，精确知道模板里哪些动态槽位不同，只补丁那些。回收是其 snapshot 模型的一等产物。

vue-lynx 没有逐模板的 hydrate 原语，于是 hydrateListItemEntry() 在运行时结构性地做：建立 element→id 反向索引，同步遍历 donor 与 target 子树，逐节点交换 textContent/属性并互换 element-id 归属。对同构 cell 正确，每次回收 O(树) 开销，且当同 reuse-identifier 的两个 cell 形状不同（如 v-if 分支）时会脆弱。见 PR #307 评审。

REPL 可执行：一个最小回收池

main-thread.js · 自复用 + 池L0
// 最小层级 4 草图：按 reuse-identifier 池化，优先自复用。
const signMap = new Map();               // uiSign -> { el, key }  (活跃)
const pool    = new Map();               // reuseId -> Map(uiSign -> { el, key })

function componentAtIndex(list, listID, cellIndex, operationID) {
  const reuseId = reuseIdOf(cellIndex);
  const bucket  = pool.get(reuseId);
  let entry;
  if (bucket && bucket.size) {                     // 复用池中子树……
    const [sign, pooled] = bucket.entries().next().value;
    bucket.delete(sign);
    hydrate(pooled.el, cellIndex);                // ……只补丁差异部分
    entry = pooled;
  } else {
    entry = { el: buildCell(cellIndex), key: keyOf(cellIndex) }; // 全新挂载
  }
  __AppendElement(list, entry.el);
  const sign = __GetElementUniqueID(entry.el);
  signMap.set(sign, entry);
  __FlushElementTree(entry.el, { triggerLayout: true, operationID, elementID: sign, listID });
  return sign;
}

function enqueueComponent(list, listID, sign) {   // cell 滚出屏幕
  const entry = signMap.get(sign);
  if (!entry) return;
  signMap.delete(sign);
  const reuseId = reuseIdOfKey(entry.key);
  if (!pool.has(reuseId)) pool.set(reuseId, new Map());
  pool.get(reuseId).set(sign, entry);             // 停放，不销毁
}
// hydrate() 是关键——它对每种框架意味着什么，见下方说明。

回收相关 PAPI 与生命周期

为何 __UpdateListCallbacks 必须每次 flush 都触发

回调闭包捕获了 item 表。在 snapshot 模型里必须刷新闭包，原生拉取才能看到新数据——故 ReactLynx 每次 flush 重绑。而 vue-lynx 的闭包读取实时 Map，重绑更接近「为对齐而对齐」而非功能必需（PR #311 评审）——这是个虽小却说明问题的信号：契约是围绕 snapshot 模型塑形的。

第二部分 · 5契约对框架的真实要求

把 PAPI 剥到只剩它对上层框架的能力要求。一个能把 <list> 适配好的框架，须提供（大致按难度排序）：

  1. 拉取式、可按索引寻址的渲染。按需、乱序地产出第 N 行的子树。多数声明式框架是自顶向下渲染整棵子树，而非孤立地渲染某个带索引的子节点。
  2. 稳定、可计算的顺序 diff。把「数据从顺序 A 变到顺序 B」转成基于 key 的 insert/remove/move——最好能从框架本就要做的 reconciliation 里免费得到。
  3. 廉价的重定向 / hydrate 原语。把一棵为 item X 建的子树、只改差异部分变成显示 item Y——这是跨 item 回收的核心。
  4. 跨线程边界的稳定标识。框架节点与原生 uiSign 之间、能挺过移动/池化/重定向的持久映射。

其中第 2 和第 3 项最有区分度：它们恰是 snapshot/带 key 的 VDOM reconciliation 模型的副产品，也恰是其它模型必须外挂的东西。

第二部分 · 6React / Vue / 信号 与契约对照

ReactLynx 天然契合

snapshot 模型 + 带 key 的 reconciliation。契约与它是共同设计的。

收益
  • reconciler 本身就是 diff——insertBefore/removal 记录是免费副产品。
  • 逐 snapshot 编译生成的 hydrate() → 精确、廉价的跨 item 重定向。
  • snapshot 实例天然就是带稳定标识的池条目。
代价
  • 需要整套 snapshot 编译器 + 运行时；并非轻量适配层。
  • 回调捕获数据 → 必须每次 flush 重绑。

Vue (vue-lynx) 可行，但有缝

在响应式 VNode 树上的自定义渲染器；无编译期 snapshot。

收益
  • 响应式让数据源那一侧变得轻松且细粒度。
  • 自定义渲染器能干净地拦截 insert/remove 写入 item 表。
  • 实时 Map 闭包规避了部分重绑压力。
代价
  • 没有免费的顺序 diff → 必须用快照上的 LIS 重建（#293）。
  • 没有 hydrate 原语 → 运行时结构性属性遍历，O(树) 且对形状脆弱（#307）。
  • 池化后的标识重映射需手工维护（重建反向索引）。

信号（类 Solid） 很有前景

细粒度响应式、编译模板、显式 <For>。

收益
  • 逐 cell 的响应式作用域几乎 1:1 映射到逐 cell 拉取 + 重定向。
  • cell 即一个响应式计算；「hydrate」=换掉行的 signal，DOM 自更新。这是最廉价的跨 item 复用。
  • <For> 已为 reconciliation 维护带 key 的标识。
代价
  • 顺序变化仍须表达为同样的 insert/remove diff。
  • 拉取式、乱序挂载与常规自顶向下创建相冲突。

Svelte / 编译型 喜忧参半

编译出的外科式更新、带 key 的 {#each} 块。

收益
  • 编译出的 update 函数给出类 hydrate 的精确重定向，类似 Solid。
  • 带 key 的 each 块承载标识。
代价
  • 面向「块」而非按索引寻址——逐 cell 拉取需适配。
  • 组件边界被编译掉，池条目更难界定。
四者共同的规律

区分「天然契合」与「可行但有缝」的唯一能力，是逐模板的重定向/hydrate 原语。编译模板的框架（React-snapshot、Solid、Svelte）免费拥有它、能精确回收；解释运行时树的框架（Vue 的 VDOM）必须结构性地模拟它——这正是 vue-lynx #307 风险的所在。

第二部分 · 7 · 方向 A哪种框架最适合当前的 <list> API？

就现状而言，这套 API 奖励「编译期模板 + 带 key reconciliation」模型——而 ReactLynx 最契合，因为二者是一起设计的。契约的每一条硬性要求（免费顺序 diff、编译器 hydrate、snapshot 池条目、每次 flush 重绑）都对上 ReactLynx snapshot 运行时已有的特性。这并非中立巧合：update-list-info 的 remove+insert 移动语义、reuse-identifier 的池键、以及 flush 即重绑的生命周期，读起来都像是从 snapshot reconciler 里抽取出来的。

不过理论上最优雅的契合，是细粒度信号（Solid 风格）：当 cell 是一个响应式计算，跨 item 回收就坍缩为「重赋该行的源 signal，让 effect 去补丁 DOM」——无需结构遍历、无需手工标识重映射，甚至比 ReactLynx 的编译 hydrate 还便宜。信号契合拉取式回收的精神，胜过 React 契合当前 API 的字面。

Vue 天生处在中间。响应式让数据源那半边很愉快，但运行时 VDOM 既没有免费 diff 也没有 hydrate 原语，于是适配层必须把两者都重建——LIS diff 与结构性属性遍历。它确实可行（PR 栈证明了主路径），但那些缝（#307 的 O(树) 脆弱 hydrate、#311 语义冗余的重绑）正是把「运行时树框架」适配到「snapshot 形状契约」上所付的直接代价。

第二部分 · 8 · 方向 B如何重新设计 <list> API 以更好支持框架？

当前契约把某一种实现（snapshot reconciler）的内部细节泄漏到了边界上。一套更「框架中立」的契约，应让每个框架贡献它擅长的那块、并停止强求它不擅长的那块。具体地：

今日 API 的摩擦受害者重设方向
框架必须提供顺序 diff（update-list-info），即便原生也在跟踪顺序、甚至在 layoutcomplete 里发出 diffResult。非 reconciling 框架（Vue、命令式）。接受一份当前顺序的带 key 快照，让原生自己算 move-diff；把 diffResult 变成一等的输入通路，而不仅是遥测。
跨 item 回收假定存在 hydrate(old,new)；没有它的框架只能伪造。运行时树框架（Vue）。增加显式的 recycleComponent(list, uiSign, newCellIndex) 回调，让框架用自己的更新路径重定向子树——不强加结构遍历。
回调捕获数据 → 必须每次 flush 重绑。所有人（无谓抖动）。把当前数据作为参数传给拉取（componentAtIndex(list, listID, idx, opID, ctx)），让回调保持标识稳定、重绑变罕见。
标识是原始的原生 uiSign，框架必须映回自己的节点。所有人。让框架在 uiSign 旁注册一个不透明句柄，原生在 enqueue/回收时原样回传——无需重建反向索引。
platform info 是「字符串型」并揉进 insertAction。类型安全的框架。用一个与 diff payload 分离的、带类型的 ListItemInfo 结构。

统一原则：边界应当向框架索取数据 + 标识，并让框架用自己的机制去执行重定向——而不是要求它复现某一特定 reconciler 的 diff 与 hydrate。今天的契约在说「你得是个 snapshot reconciler」。一套框架中立的契约会说「把你的 key 和一个更新 cell 的办法给我，我来驱动」。这个改动会让 Vue 适配层同时甩掉它的 LIS diff 与结构性 hydrate，会让信号类框架把逐 cell 的 effect 直接插进来，而对 ReactLynx 几乎零成本（它若愿意，仍可在内部自己算 diff 和 hydrate）。

结论

方向 A：当前 API 最契合 ReactLynx（源于共同设计），而对 Solid 式信号最优雅；Vue 可行，但恰在 PR 栈不得不绕开的那两个点上，为阻抗失配付费。方向 B：把顺序 diff 与标识跟踪下沉给原生，并用显式的 recycleComponent 重定向回调替换隐式的 hydrate 假定——把一个「snapshot 形状的契约」变成「数据 + 标识的契约」，让任何带 key 的框架都能用自己的更新路径满足它。
