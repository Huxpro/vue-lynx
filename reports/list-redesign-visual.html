A framework-neutral <list>: three operations, and everything else optional 一个框架中立的 <list>：三个操作，其余皆可选

The current <list> contract asks every framework to be a snapshot reconciler. It doesn't have to. Below: an interactive model of what the list actually needs, and a redesign that asks frameworks only for data + identity — letting each keep its own way of updating a cell. 当前 <list> 契约要求每个框架都去当一个 snapshot reconciler。其实不必。下面是一个可交互模型，展示 list 真正需要什么，以及一套只向框架索取数据 + 标识的重设方案——让每种框架保留自己更新 cell 的方式。

What to notice. With recycling on, “DOM cells created” plateaus at roughly window + pool no matter how far you scroll. Turn it off and it climbs without bound — every scroll destroys and rebuilds. Recycling lives entirely on the framework side — which is exactly why it should not be a shared primitive. 观察点。回收开启时，无论滚多远，「创建的 DOM cell」都稳定在「窗口 + 池」附近。关闭后它无上限地增长——每次滚动都在销毁并重建。而两种情况下原生的协议完全一样：它只调用了 mount / release。回收完全活在框架侧——这正是它不该成为共享原语的原因。

The keystone拱心石

Because native pulls, the framework can never “render the list.” It can only be a data source that answers questions out of order. Every design choice below is judged by one test: does it respect these four, or does it smuggle in a fifth assumption? 因为原生 pull，框架永远无法「渲染整个 list」。它只能是一个能乱序应答的数据源。下面每个设计选择都用一条标准来评判：它是尊重这四条，还是偷偷夹带了第五条假设？

Layer 1 · the minimal contract第 1 层 · 最小契约A cell's whole life — and where recycling actually hides一个 cell 的一生——以及回收究竟藏在哪

Trace one cell. The only calls that cross the boundary are mount and release. The fork after release — destroy vs park — happens inside the framework. Native never learns which branch was taken.追踪一个 cell。跨越边界的调用只有 mount 和 release。release 之后的分岔——销毁还是停放——发生在框架内部。原生永远不知道走了哪条分支。

idle空闲 ━▶mount() VISIBLE可见 ━▶release() parked (pool)停放（池） ↺mount()/retarget() VISIBLE可见 ━▶or destroyed销毁

The dashed parked → visible loop is the entire recycling story. It needs exactly two things from native, and nothing else:停放 → 可见这条虚线回路，就是回收的全部故事。它只需要原生给两样东西，别无其它：

  • Native must accept a reused token. mount may return a token native has seen before — it can't assume every mount is a fresh element.原生必须接受被复用的 token。mount 可以返回一个原生见过的 token——它不能假设每次 mount 都是全新元素。
  • An optional reuseClass hint so native can size its own container buffers. The framework may ignore it entirely.一个可选的 reuseClass 提示，好让原生给自己的容器缓冲定容量。框架可以完全无视它。

Layer 2 · conflict vs freedom第 2 层 · 冲突 vs 自由What frameworks must bend to — and what the API wrongly locks框架必须屈服于什么——以及 API 错误锁死了什么

Map every design concern on two axes: is it fundamentally forced by the model (right), or an incidental degree of freedom (left)? And does today's API lock it (bottom) or leave it free (top)? The problem cases are the red ones in the bottom-left: real freedoms, needlessly locked.把每个设计关注点放到两根轴上：它是被模型根本强制的（右），还是一个偶然的自由度（左）？今天的 API 是锁死它（下）还是放行（上）？问题就在左下角那些红点：本是自由，却被无谓地锁死。

legit constraint (keep)正当约束（保留） freedom wrongly locked (fix)被错锁的自由（修） deliberate gate (worth it)刻意的门（值得）

The three red dots are locked by a single root cause: the contract assumes a hydrate(old,new) exists and makes callbacks close over data. Remove those two assumptions and all three float up into “free” — no framework is forced into a VDOM shape, a reconciler, or a particular thread split.这三个红点被同一个根因锁住：契约假定存在 hydrate(old,new)，并让回调闭包捕获数据。去掉这两个假设，三者都会浮上「自由」区——没有框架被迫套进 VDOM 形状、被迫自带 reconciler、或被迫某种线程切分。

The redesign · capabilities重设 · 能力分层One required core, three optional capabilities一个必需内核，三个可选能力

Instead of one monolithic interface everyone must fully implement, layer it. The core is the Layer-1 contract. Each capability lets a framework contribute the thing it is good at — and skip what it isn't.不再是一个所有人都得完整实现的单体接口，而是分层。内核就是第 1 层契约。每个能力让框架贡献它擅长的那块——并跳过不擅长的。

required core · ListSource

order() · mount(key) · release(token)

Enough to run, with no recycling. Native diffs order() itself. Implementable by imperative JS in ~10 lines.足以运行，无需回收。原生自己 diff order()。命令式 JS 十行即可实现。

optional capability · Retargetable

retarget(token, key) — replaces the imposed hydrate

The framework re-points an existing cell to a new key via its own update path: signal reassign, compiled patch, or VDOM patch. Absent → native falls back to release()+mount(). This one hook is what un-privileges the snapshot model.框架用自己的更新路径把已有 cell 重指向新 key：signal 重赋、编译 patch、或 VDOM patch。缺省 → 原生回退到 release()+mount()。正是这一个钩子，让 snapshot 模型不再享有特权。

optional capability · SelfDiffing

mutations() — reconcilers keep their small payload

A framework that already computes a keyed diff (ReactLynx) may hand it over directly instead of a full order() array — saving payload on huge lists. React loses nothing; everyone else ignores it.已经在算 keyed diff 的框架（ReactLynx）可以直接把它交出来，而非传完整 order() 数组——在超大列表上省 payload。React 分文不损；其余框架直接无视。

optional capability · AsyncMountable

mountAsync(key) — defer, with operationID confined here

Only when declared does native thread the operation-correlation ids that today pollute every path. The synchronous 90% stays clean.只有声明它时，原生才会穿针引线那套「操作关联 id」——今天它污染了每一条路径。同步的 90% 保持干净。

type CellKey = string;
type CellToken = number & { readonly __brand: 'CellToken' };  // opaque, native round-trips it

// —— required: give data + identity, that's all ——
interface ListSource {
  order(): readonly CellKey[];                 // native diffs this into moves
  describe?(key: CellKey): ListItemInfo;         // typed, not stringly platform-info
  mount(key: CellKey, hint: { reuse?: string }): CellToken;
  release(token: CellToken): void;             // destroy OR park — native doesn't care
}
// —— optional capabilities, composed à la carte ——
interface Retargetable  { retarget(token: CellToken, key: CellKey): void; }
interface SelfDiffing   { mutations(): ListMutations; }
interface AsyncMountable{ mountAsync(key: CellKey, hint: object): Promise<CellToken>; }

Is it worth it?值得吗？

Yes, and asymmetrically so. Every freedom-increasing move (native diffs order(), opt-in retarget, data-by-argument) is paired with an escape hatch that costs the incumbent nothing — ReactLynx keeps mutations() and its compiled hydrate. The freedom-decreasing moves (mandatory keys, gated async, sealed cell tree) are deliberate constraints that buy soundness and a clean common path. Net: Vue sheds its two worst seams, signals plug straight in, and React is untouched. 值得，而且是不对称地值得。每一个「增加自由度」的动作（原生 diff order()、可选 retarget、数据走参数）都配了一个对既有者零成本的逃生口——ReactLynx 保留 mutations() 和它的编译 hydrate。而「减少自由度」的动作（强制 key、gated 异步、封住 cell 树）是刻意的约束，换来正确性与一条干净的公共路径。净效果：Vue 甩掉两条最糟的缝，signals 直接插入，React 毫发无损。
